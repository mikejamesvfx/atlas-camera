This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

For the artist/TD-facing explanation of camera recovery, matte-painting projection,
and preview dilation, see [docs/USER_GUIDE.md](docs/USER_GUIDE.md). For the full
ecosystem map (complete node catalog, geometry-derivation strategies, multi-angle

## Commands

```powershell

pip install -e ".[dev]"

# Run all tests
python -m pytest -q

# Run a single test file
python -m pytest tests/test_intrinsics.py -v

# Run a specific test
python -m pytest tests/test_intrinsics.py::test_focal_length_to_pixels -v

# Install with UI and vision support

# Install neural extras (GeoCalib learned prior + Depth Anything V2) — needed

# Install atlas_camera into ComfyUI's venv (editable — run once)

# Verify ComfyUI can import the package
& "<COMFYUI_ROOT>\venv\Scripts\python.exe" -c "import atlas_camera; print(atlas_camera.__file__)"
```

## Optional dependency groups

- `dev` — numpy, opencv-python, pytest
- `vision` — numpy, opencv-python (runtime vision)
- `image` — Pillow only
- `usd` — usd-core
- `ui` — FastAPI, uvicorn, Pillow, python-multipart
- `neural` — torch + GeoCalib (learned single-image camera prior). GeoCalib is GitHub-only: `pip install "git+https://github.com/cvg/GeoCalib.git"`. torch is expected from the host env (e.g. ComfyUI's venv).

The core package has **zero required runtime dependencies**. All vision, USD, and UI imports are guarded with informative `ImportError` messages.

## Architecture

```
atlas_camera.core       ← DCC-agnostic schema, solver, math (no host deps)
atlas_camera.exporters  ← Maya, Blender, Nuke, USD, review package writers
atlas_camera.importers  ← Atlas JSON and USD camera loaders
atlas_camera.comfy      ← ComfyUI node library (25 nodes, no hard Comfy dep)
atlas_camera.ui         ← Optional FastAPI project service
atlas_camera.reference_data ← Curated scale-reference registry (JSON)
atlas_camera.gaussian   ← Future 3DGS / point-cloud interfaces (placeholder)
atlas_camera.inference  ← Optional local multimodal provider helpers
ui/                     ← React/Vite workbench (Three.js 3D viewport)
examples/               ← Example ComfyUI workflows and test images
```

The public API is `import atlas` (thin facade in `atlas_camera/__init__.py`). The stable package name is `atlas_camera`.

## ComfyUI integration (`atlas_camera/comfy/`)

### Setup

A symlink connects the node pack to ComfyUI's custom_nodes directory (paths below
are this machine's; substitute your own ComfyUI install and repo checkout locations):

```
<COMFYUI_ROOT>\custom_nodes\AtlasCamera
    → <REPO_ROOT>\atlas_camera\comfy
```

The package is installed in editable mode in ComfyUI's venv so `import atlas_camera` resolves to this project directory. Changes to Python source are live immediately — no reinstall needed.

### Double-import guard (critical)

`atlas_camera/comfy/__init__.py` is loaded twice at ComfyUI startup:
1. As `AtlasCamera` custom node (by ComfyUI's node loader)
2. As `atlas_camera.comfy` package (by `from atlas_camera.comfy.nodes import ...` inside the same file)

Both loads hit the same file, causing the aiohttp route `GET /atlas/camera_data/{node_id}` to be registered twice, which raises `RuntimeError: method HEAD is already registered`. The fix in `__init__.py` checks `if not any(r.path == ... for r in _routes)` before registering.

### Frontend extension (`atlas_camera/comfy/web/atlas_blockout.js`)

Registers as `AtlasCamera.Blockout` ComfyUI extension targeting `AtlasBlockoutViewport` nodes. On node creation it builds a Three.js canvas with a **self-contained orbit controller** (`createOrbitControls` — the examples/jsm `OrbitControls` uses a bare `import ... from "three"` that browsers can't resolve without an import map, so it never loaded; the custom one depends only on the already-loaded THREE), a primitive toolbar (Box/Plane/Cylinder/Person/Clear), scale-reference proxy buttons (🧍 Woman / 🚗 Sedan), a 📷 Camera View reset, and a Render Passes button.

The viewport **inherits the recovered camera**: `applyRecoveredView()` sets the Three.js camera to the recovered pose/fov, then initialises the orbit controller *from* it (`syncFromCamera`, pivot = the looked-at ground point) so the default view matches the source photo; dragging orbits, and 📷 Camera View snaps back. For exact photo alignment set the node's `width`/`height` to the source image's aspect ratio (the camera aspect = `target_width/target_height`). The background photo is a plane sized to fill the recovered frustum along the view axis.

**Camera projection (📽 Project, matte-painting mode):** the payload's `proxy_geometry` entries (from `AtlasDeriveProjectionGeometry`) are built as meshes (`buildDerivedProxies`, group `atlas_derived_proxies`, transforms fed verbatim to `Matrix4.set` — row-major both sides). `makeProjectionMaterial` is the GLSL port of `ui/src/ProjectionMaterial.ts` (world pos → recovered-camera pixel → sample source photo; discard behind camera / outside frame) with `depthWrite:true` (multi-proxy occlusion; the ui original was a single-ground overlay) and a **flipY:false** texture (top-left UV origin — never share the background texture, which uses default flipY). The 📽 toggle swaps ALL projectable meshes (derived + user primitives + OBJ proxies) between grey and the shared projection material; `_prevMaterial` is stashed only once so material rebuilds don't lose the original. Clear leaves the derived group alone (Python-owned; regenerates each execution). **Matte-painting property:** texels are assigned by ray, so geometry at slightly-wrong depth still receives exactly the pixels its silhouette subtends — perfect reassembly from Camera View; scale error only shows as parallax when orbiting.

**🎬 Backdrop toggle:** every primitive-fitting derivation strategy (`azimuth_walls`, `vertical_extrusion`, `ransac_planes`, `room_cuboid` — never `relief_mesh`) always emits one extra flat `"projection_backdrop"` plane (`proxy_geometry.py`/`plane_extraction.py`/`room_layout.py`, all via the shared `depth_geometry.build_backdrop_primitive`) sized to cover the whole frustum at the far-depth percentile — a catch-all so 📽 Project never shows raw background behind the fitted primitives. When `geometry_mode="both"` this backdrop plane is also projectable and sits behind/around the actual relief mesh, receiving its own copy of the projected texture. The toggle just sets `.visible` on any mesh named `"projection_backdrop"` (`scene.traverse`, matched by name, not a stashed reference — a plain visibility flag handles both the grey preview AND 📽 Project identically, since an invisible mesh never renders regardless of material), so Project can be limited to painting only the generated mesh. Re-applied inside `setProxies()` after every `buildDerivedProxies()` call, since that rebuilds fresh mesh objects (default `visible=true`) on each execution — without the reapply, toggling it off would silently reset on the next Queue.

**Viewport diagnostics — exposure, VP/horizon/ground diagram, camera HUD (added 2026-07-02):** three toolbar additions, all pure frontend (no new node widgets, payload-only backend change).
- **☀ Exposure** — `renderer.toneMapping = THREE.ACESFilmicToneMapping` + a slider controlling `renderer.toneMappingExposure` (0.1–3, default 1). Only ever affects the LIT grey (`MeshStandardMaterial`) preview: the projection `ShaderMaterial` writes `gl_FragColor` directly with no tone-mapping GLSL chunk (immune by construction), and `renderAllPasses`' normal/mask override materials are explicitly `toneMapped:false` so the exposure slider can never corrupt those deterministic passes (the custom depth shader is likewise immune — no tonemapping chunk).
- **📊 Diagram** — layered SVG overlay (absolutely positioned over the canvas, `pointer-events:none` so it never blocks orbit dragging) with 3 independently-opacity-dimmable layers: VP fan-lines (image-corner-to-VP-position lines + labeled marker per vanishing point, colored orange/blue/green for left/right/vertical matching `AtlasVPVisualization`'s PIL scheme), horizon (line + confidence label), ground (shaded rect below the horizon split). viewBox uses the solve's native image pixel dimensions, so VP/horizon positions from the payload need no rescaling. **VPs are empty on the learned (GeoCalib) solve path** — it predicts focal+gravity directly, never via classical vanishing points — the layer only populates on the `detect_vanishing_points=True` VP path; horizon/ground work on both. Off-canvas VPs (common — e.g. `(-2297,-392)` on a real test image) are simply clipped by the SVG's default `overflow:hidden`, leaving the converging fan-lines visible at the frame edge.
- **ℹ Info** — HUD panel (top-left, monospace, semi-transparent) showing solved lens (focal mm + FOV°), sensor mm, camera height m, scene depth m (from the backdrop's `distance_m` if derived), confidence %, source method, scale tier — each line only rendered when its value is non-null.
- Backend: `_extract_blockout_camera` gained `vanishing_points` (list of `{position_px, direction_label, confidence}`), `horizon_line` (`{endpoints_px, line_coefficients, confidence}`), and `camera_meta` (`{confidence, source_method, scale_source, focal_mm, sensor_mm, fov_h_deg, camera_height_m, scene_depth_m}`) — all pulled from data already on the solve, no new computation beyond `fov_h_deg` and reading the backdrop primitive's `distance_m`.

**Orbit coverage (why the mesh/projection can look like it "disappears" on rotate):** derived geometry only ever covers what the recovered camera could see — a forward-facing cone, inherent to single-photo reconstruction. `createOrbitControls` clamps yaw/pitch to ±80°/±55° around the recovered camera's own direction (`theta0`/`phi0`, wraparound-safe via `atan2(sin,cos)`, re-anchored on every `syncFromCamera()` — i.e. every execution and every 📷 Camera View click) so you can't orbit past the reconstructed cone into empty space in the grey/undressed preview.

**Orbit pivot — geometry centroid, not a ground heuristic:** `applyRecoveredView` sets an initial pivot via `groundPointInView` (where the camera's forward ray crosses Y=0, capped at the solved scene depth) purely as a fallback for before any geometry exists (or for workflows that never run `AtlasDeriveProjectionGeometry` at all) — it's a generic heuristic, not the actual centre of whatever gets reconstructed, and could be visibly off for reconstructions that aren't roughly centred on the camera's exact forward ray. Once `buildDerivedProxies` has real geometry, `setProxies` immediately overrides it with `computeGeometryPivot()` — a `THREE.Box3` bounding-box centroid over every mesh in the `atlas_derived_proxies` group, deliberately excluding any mesh named `"projection_backdrop"` (the same 🎬 Backdrop plane, above — including it would drag the centroid out toward the frustum's far edge instead of the actual reconstructed subject). This only calls `controls.setTarget(...)` + `syncFromCamera()` (re-deriving the orbit sphere from wherever the camera *already* is) — it never moves the camera itself, so recomputing it on every execution (including re-executions from ⏺ Bake Path) can't disrupt an in-progress inspection, and is a no-op in effect when the geometry hasn't changed between runs.

`AtlasBlockoutViewport`'s `preview_expand` widget (default 1.0 = off) can optionally dilate the geometry itself for more coverage within that arc, via `proxy_geometry.dilate_proxy_geometry_for_preview()`: for any point with local normal n̂, radiating from the camera position, `p' = pivot + ((p-pivot)·n̂)n̂ + scale·[(p-pivot) - ((p-pivot)·n̂)n̂]` — only the normal-perpendicular offset scales, so a plane widens without drifting in depth, a box/cylinder (no single normal) dilates uniformly, and the relief mesh dilates per-vertex using each vertex's own (genuinely arbitrary) normal. Applied ONLY at `serialize_proxy_geometry()` time (`_extract_blockout_camera` passes `preview_expand`/`preview_pivot=camera_position`) — never mutates the `AtlasProxyPrimitive` objects on the solve, so DCC exports and metric measurements stay accurate regardless of the viewport setting.

**Known interaction, not further fixable without dual geometry (dilation vs. 📽 Project):** dilation and the projection shader are fundamentally in tension, and this is *not* the same bug the clamp fixes. The 📽 Project shader assigns texels by casting each fragment's world position through the RECOVERED (undilated) camera and discarding anything that lands outside the actual photographed frame (`atlas_blockout.js`'s `PROJECTION_FRAGMENT_SHADER`, `discard` on out-of-bounds/behind-camera). Dilated geometry is, by construction, surface area the camera never actually photographed — there is no real pixel data for it, so it correctly discards to empty/black. Because dilation is most visible away from dead-center, this shows up as large black regions after only a *moderate* orbit with Project active — confirmed via live reproduction: with the prior default (`preview_expand=1.4`), a 60px drag left roughly half the frame black; the identical drag at `preview_expand=1.0` showed full coverage (parallax-shifted, as expected). This is why the default was changed to 1.0 — raise it only when inspecting undressed blockout primitives, never while relying on 📽 Project.

Proxy meshes live in `examples/models/*.obj` (authored in **centimetres**; the loader scales by 0.01 into the metric world). Loaded via Three.js `OBJLoader` from `GET /atlas/proxy_model/{name}` and dropped on the ground (Y=0) under the camera's view centre — a correctly-sized human/car is the fastest visual check that the solve + camera height are right. Loaded groups are tagged `userData.atlasProxy` so Clear removes them.

On each `node.onExecuted`: fetches `GET /atlas/camera_data/{node_id}` to receive the recovered camera dict (`view_matrix`, `fx`, `fy`, `source_image_b64`) and applies it to the Three.js camera. The source image is loaded as a background plane from `cameraData.source_image_b64`.

`Render Passes` encodes four WebGL passes (shaded, depth, normal, mask) as base64 PNG, writes them as JSON into the `client_data` STRING widget, then calls `app.queuePrompt(0, 1)` to send them back to Python.

## Solve entry points

- `atlas.recover(image_path, method="vanishing_points"|"learned", ...)` → `LatentScene` — primary API
- `atlas_camera.core.solver.solve_still_image_learned(image_path, camera_height="auto"|float, depth_model=..., ...)` — learned GeoCalib prior (needs `[neural]`); robust on AI-generated images. `camera_height="auto"` measures height from depth instead of assuming it
- `atlas_camera.core.solver.estimate_ground_height_from_depth(depth, rotation=..., fx, fy, cx, cy)` — pure-numpy ground-plane fit → camera height + confidence + ground mask
- `atlas_camera.core.relief_mesh.build_relief_mesh(depth, view_matrix=..., fx, fy, cx, cy, ...)` — triangulated/decimated depth mesh, torn at silhouettes, camera projection baked into UVs; `estimate_ground_scale(...)` pins the ground to Y=0. Exported via `exporters.relief_mesh_exporter.export_relief_mesh()` (OBJ+MTL+texture) or `export_relief_mesh_glb()` (self-contained GLB, zero-dep glTF 2.0 writer, texture embedded, KHR_materials_unlit; glTF is Y-up like Atlas — coordinates pass through, only the UV V origin flips)
- `atlas_camera.core.solver.metric_height_from_reference(base_px, top_px, real_height_m, rotation=..., fx, fy, cx, cy)` — single-view metric camera height from one known-size vertical object
- `atlas_camera.core.solver.resolve_reference_scale(refs, ...)` / `apply_reference_scale(solve, refs)` — aggregate multiple reference objects; rescale a solve in place (tier-1 metric scale)
- `solve_still_image_learned(..., scale_references=[...])` — reference objects are tier-1 scale, above depth (tier-2) and the assumed default (tier-3)
- `atlas_camera.inference.depth_estimator.estimate_depth(image_path, model_id=...)` — Depth Anything V2 monocular depth (metric or relative)
- `atlas_camera.core.solver.solve_from_learned_prior(prior, ...)` — pure-numpy builder from a `CameraPrior` (keeps torch out of core)
- `atlas_camera.inference.learned_prior.estimate_camera_prior(image_path)` — run GeoCalib, returns a torch-free `CameraPrior`
- `atlas_camera.core.solver.solve_from_constraints(image_path, constraints_dict)` — artist-guided line constraints
- `tools/solve_image.py` — CLI: auto VP detection + debug overlay + review package
- `tools/solve_constraints.py` — CLI: JSON constraints → review package

## Key design rules

- Core schema is pure Python dataclasses — no external deps.
- Optional deps (`numpy`, `cv2`, `fastapi`, `pxr`) are always guarded by try/except with `pip install -e .[extra]` hints in the error message.
- Adapter boundaries must be explicit: coordinate-system conversions are never silent.
- `LatentComponent` slots (`depth`, `geometry`, `lighting`, `semantics`) default to empty until their solvers exist — review packages describe unsupported components rather than omitting them. `depth` is now populated by `solve_still_image_learned(camera_height="auto")` (Depth Anything V2 map summary + measured camera height + confidence); `geometry`/`lighting`/`semantics` remain empty.
- **View-matrix convention (critical):** geometry/camera math must use the full 4×4 `extrinsics.camera_view_matrix` end-to-end (`cam_to_world = inv(view_matrix)`, row-major, column-vector points, translation in column 3) — the convention proven to match the viewport shader. Never build world math from the 3×3 `camera_rotation_matrix` (it has a transpose ambiguity). `core/proxy_geometry.py`, `core/depth_geometry.py`, `core/plane_extraction.py`, `core/room_layout.py`, and `_ground_depth_compute` all follow this; `THREE.Matrix4.set()` consumes the row-major floats verbatim.
- **Multiple geometry-derivation strategies, always artist-selected, never auto-picked:** `AtlasDeriveProjectionGeometry`'s `geometry_mode` picks the output kind — `relief_mesh` (default, a single triangulated mesh following actual depth contours — handles arbitrary/organic shapes but has real gaps at sharp depth discontinuities), `primitives` (simple fitted shapes — cleaner/blockout-like), or `both` (both at once, mostly for comparison; they visually overlap). When `primitives`/`both` is selected, `primitive_method` picks the fitting strategy: `azimuth_walls` (default — vertical walls + foreground boxes/cylinders only, general-purpose but height is clipped to whatever 3D points individually pass a near-vertical-normal filter, so it truncates at a sloped roof/spire/tower — confirmed on real church/tower photos), `ransac_planes` (any-orientation planes via sequential RANSAC seeded by a 2D orientation histogram — best for exterior architecture: roofs, ramps, stepped facades), `room_cuboid` (Manhattan floor+≤4 walls+optional ceiling aligned to the dominant wall direction — best for genuinely orthogonal interiors; produces confidently wrong/skewed results on non-orthogonal rooms, so the artist must pick the right method for the shot rather than relying on auto-detection), `vertical_extrusion` (same wall orientation/distance detection as `azimuth_walls` via the shared `proxy_geometry._cluster_walls_by_azimuth` helper, but height comes from the image-space silhouette instead — the topmost non-sky pixel per column, back-projected at its own depth regardless of local surface normal, per Hoiem/Efros/Hebert's "Automatic Photo Pop-up" (SIGGRAPH 2005) billboard-cutout technique; reaches towers/spires/sloped roofs `azimuth_walls` truncates, at the cost of representing them as a flat vertical plane rather than their true sloped shape). `core/depth_geometry.py` factors the shared back-projection/normals/ground-fit/backdrop logic (bit-for-bit consistent with `proxy_geometry.py`'s own copies, which are left untouched to avoid regression risk) so `proxy_geometry.py` (`azimuth_walls`, `vertical_extrusion`), `plane_extraction.py` (`ransac_planes`), and `room_layout.py` (`room_cuboid`) all agree on world points and metric scale for a given depth map. Every extractor's `stats` dict must include `ground_scale` — required contract with the node's relief-mesh branch, which reuses whichever method's scale was computed. Cross-product normals aren't camera-oriented by construction; `plane_extraction.py`/`room_layout.py` flip them toward the camera globally right after back-projection (sign-invariant under the later ground rescale) so orientation metadata and per-side wall assignment are meaningful.
- **Foreground object height plausibility ceiling (`ProxyDerivationConfig.object_max_height_m`, default 6.0):** `_derive_objects`' foreground-box/cylinder clustering (shared by `azimuth_walls` and `vertical_extrusion` via the same `_derive_objects` call) groups candidate points into objects using a **2D (X,Z) occupancy grid** — it has no notion of height when deciding what belongs to the same object. Found via live browser verification (not unit tests, which all used clean synthetic scenes): on a real photo, a genuine foreground object's footprint spuriously merged with an unrelated tall background structure behind/above it (one that wall-fitting hadn't already claimed) because they shared the same X/Z occupancy cells — the resulting box's height (98th-percentile Y) reflected the tall structure, not the actual foreground object, producing an implausible ~14m "foreground" box that filled the entire viewport edge-to-edge. Mirrors the same "reject an implausible fit" pattern `_fit_cylinder` already uses for `cylinder_radius_range` (returns `None` outside the plausible radius band): when a cluster's fitted height exceeds `object_max_height_m`, refit using only the points at or below that ceiling (dropping the object entirely if too few remain) rather than passing an unbounded height through. Applies to both `derive_projection_proxies` and `derive_vertical_extrusion_proxies` since both call the same shared `_derive_objects`. Does not fix the underlying 2D-only clustering (a genuinely depth/height-aware clustering rewrite is out of scope here, and carries real regression risk for currently-working cases) — this is a plausibility guard on the *output*, not a fix to the clustering algorithm itself.
- **Sky-aware depth (`depth_geometry.detect_sky_mask`):** monocular depth (Depth Anything and similar) has no reliable cue on feature-less sky/cloud regions and often hallucinates noisy, spatially-incoherent depth there rather than a smooth "far" value — left unhandled, that noise gets triangulated into a jagged, spatially-huge mesh chunk (`relief_mesh.build_relief_mesh`) that dwarfs the actual photographed geometry, and can even leak into wall/primitive fitting. `detect_sky_mask(depth, horizon_y=...)` flags a pixel as sky when it's above the solved horizon line AND either at/beyond a far-depth percentile or has high local **roughness** — mean-squared discrete Laplacian, not raw variance — relative to the below-horizon baseline. Roughness (not variance) is deliberate: a genuinely sloped surface (a real roof, a ramp) has a constant local gradient, so its Laplacian is ~0 even though its raw variance in a window is large; plain variance would misclassify real sloped architecture as sky. `build_relief_mesh` uses this to exclude sky from triangulation entirely (a hole, like any other depth discontinuity) rather than merely distance-clamping it; `vertical_extrusion` (above) uses the same mask to find each wall's real silhouette top instead of guessing from noisy sky depth.
- **Multi-angle patch projection (`AtlasAddPatchView` + `ProjectionSource` + `camera_math.orbit_camera`):** single-photo projection only textures what the recovered camera saw; occluded/grazing areas go black on orbit. To fill them, an AI novel view of the same scene (generated at a defined angle, e.g. the Qwen-Image-Edit-2511 Multiple-Angles LoRA, `<sks> [azimuth] [elevation] [distance]`) is added as a **patch**. The crux is registration: the LoRA output is an independent generation with no ground-truth transform back to the scene, so the patch camera is **constructed, not solved** — `orbit_camera(primary_extrinsics, pivot, d_azimuth, d_elevation, distance_scale)` orbits the recovered camera around the scene pivot (its ground look-at point, `ground_lookat_pivot`) and rebuilds the view matrix via an unambiguous `look_at` (so it shares the primary's Y-up right-handed world→cam frame by construction). **The LoRA angles are ABSOLUTE (subject-relative), not relative to the source view** — "right side view" is 90° around the *subject's* front, matching the ComfyUI-qwenmultiangle named views (0/45/…/315 azimuth, −30/0/30/60 elevation, close/medium/wide). So `AtlasAddPatchView` takes both the source's named view and the patch's named view and applies the **difference** as the orbit (`patch − source`); a straight-on source is "front view" so the patch's named angle maps directly. GeoCalib can't recover the azimuth delta automatically (single-image yaw is unobservable — see `solver._rotation_from_up_vector`), which is exactly why this is a named-view input plus a `flip_azimuth` handedness toggle, calibrated by eye. `AtlasAddPatchView` then derives the patch view's own relief geometry (Depth Anything) *in that constructed frame* and appends a `ProjectionSource` (schema: camera + novel-view `image_b64` + its `proxy_geometry` + priority + angle provenance; JSON-safe, backward-compatible empty default on `LatentScene`). The viewport (`atlas_blockout.js` `buildPatchSources`) builds each source's geometry with its own projection material bound to that source's camera+image, plus a **facing-ratio mask** (`uFacingThreshold`: `|normal·dir-to-camera| < threshold` discards grazing fragments — the dot-product occlusion from gs_mptk; primary passes −1 = never discards and keeps priority, patches pass ~0.2 so they only paint head-on surfaces and fall through elsewhere). The facing mask decides *whether* a source can paint a fragment at all; `ProjectionSource.priority` then decides *which* source wins where more than one can — `priorityToRenderOrder`/`priorityToOffsetUnits` (same file) convert it into a per-mesh `renderOrder` (deterministic exact-depth ties) plus a small priority-scaled `polygonOffsetUnits` bias (wins only the near-tie band; a genuinely nearer mesh's real depth still wins normal z-test), with the primary given a sentinel `renderOrder` above any patch so it stays implicitly highest without a synthetic priority number. **Load-bearing assumption, verify first:** the LoRA's requested angle must actually correspond to the constructed orbit — this is a calibration go/no-go (see `examples/atlas_camera_multiangle_patch_proof_workflow.json`), because both the camera placement and the patch-depth fusion happen in the constructed frame. gs_mptk (the old Maya Mattepainting Toolkit) is the *conceptual* reference for occlusion-masked multi-projection only; the whole pipeline here is Atlas/ComfyUI-native. The named-view dicts and the patch − source delta math (`_AZIMUTH_VIEWS`/`_ELEVATION_VIEWS`/`_DISTANCE_VIEWS`/`_named_view_orbit_delta` in `nodes.py`) are module-level, not `AtlasAddPatchView`-private, specifically so `AtlasOcclusionMask` (below) can place its own target/patch camera identically — any drift between the two would silently misalign a precomputed occlusion mask from the patch geometry `AtlasAddPatchView` derives afterward for the same image. `AtlasOcclusionMask` is a pure backend/numpy pre-pass (`depth_geometry.primary_camera_validity_mask`, no browser round-trip — deliberately, so it runs headlessly via the ComfyUI API and stays out of `atlas_camera.core`'s "no Three.js dependency" rule) that masks where the PRIMARY camera's projection is invalid at the target/patch view (behind-camera/out-of-frame/too-grazing — the same facing-ratio test as `uFacingThreshold` above, just evaluated in Python against the primary instead of in the projection shader against each patch), meant to feed `ImageCompositeMasked` before the result becomes an `AtlasAddPatchView` patch. It is Phase 1 only (frustum/frame/facing-angle); true depth-shadow occlusion (surfaces hidden behind nearer geometry from the primary's view despite passing those tests) is designed but not yet built — see `docs/atlas_occlusion_mask_implementation_plan.md`.
- **Camera path animation (`AtlasBlockoutViewport`'s 🎥 Camera Path mode + `camera_path.py` + `AtlasExportCameraPathUSD`):** single-photo projection is normally inspected from one static (orbit-clamped) viewpoint; this mode lets an artist test how it holds up under an actual camera *move* — orbit/dolly/pan — without a DCC. A camera path is a list of `AtlasCameraKeyframe` (`frame_index`, `position`, `target`, `up`, `easing`) plus `fps`/`frame_count` (`schema.AtlasCameraPath`), authored entirely client-side against the same loaded geometry/photo the viewport already has. Authoring uses a **separate, deliberately UNCLAMPED fly controller** (`createFlyControls`, RMB-hold + WASD/QE) rather than the existing orbit control — the orbit control's ±80°/±55° clamp exists specifically to keep static inspection inside the recovered camera's cone, but leaving that cone is the entire point of testing a move. `camera_path.sample_camera_path` (pure Python, no numpy/torch, mirrors `camera_math.py`'s style) turns keyframes into a full `AtlasExtrinsics` per frame via Catmull-Rom (phantom-duplicated endpoints) + per-segment easing, then `camera_math.look_at_view_matrix` — sampling exactly at a keyframe's `frame_index` always reproduces that keyframe exactly, regardless of easing (t=0/1 are fixed points for every easing function used). **What actually moves during preview/bake is the viewport's render/viewing camera — never the projection shader's `uAtlasViewMatrix` uniform**, which stays bound to whichever camera (primary or patch) actually photographed the texture; this is the same mental model the existing static orbit-and-inspect already uses (drag to view from elsewhere while the projected texture stays anchored to its real source camera), just driven by a keyframed path instead of a mouse drag. ⏺ Bake Path steps through `0..frame_count-1`, sets the render camera to each sampled pose, calls `renderer.render()`, and reads the canvas directly via `canvas.toDataURL()` (no render-target/flip-Y dance needed — unlike the depth/normal/mask passes, this is a plain "what you see is what you get" capture) into `path_frames` (an IMAGE batch feeding a Video Combine node) and `camera_path` (the raw keyframes, feeding `AtlasExportCameraPathUSD` for a time-sampled `.usda`) — both written into the same `client_data` widget/channel "Render Passes" already uses, merged rather than overwritten so the two buttons don't clobber each other's results. **Known, deliberate duplication:** the Catmull-Rom+easing math exists in both `camera_path.py` (the source of truth for Bake/USD) and `atlas_blockout.js` (`catmullRom3JS`/`sampleKeyframePoseAtFrame`, for live 60fps Play scrubbing without a Python round-trip) — the same kind of accepted duplication as `depth_geometry.py`/`proxy_geometry.py`; keep them in sync by hand. **Inherits, not a new bug:** frames sampled outside the recovered camera's cone render the same documented black/undefined regions as orbiting past it under 📽 Project (see "Orbit coverage" below) — camera FOV is intentionally NOT keyframed/interpolated in this pass (Atlas intrinsics are fixed per solve, matching every other camera-construction path here); `AtlasCameraKeyframe.fov_deg` is stored for future use but unused by today's sampler/playback. **Presets** (`computePresetEndPose` in `atlas_blockout.js`) give a one-click 2-keyframe quick-start instead of hand-placing keyframes with fly nav every time — Pan L/R (rotates the TARGET around the fixed camera position, i.e. the camera swivels in place), Orbit L/R (rotates the POSITION around the fixed target, i.e. the camera arcs around the subject), Dolly In/Out (moves the POSITION toward/away from the fixed target along the view axis). All three derive "left/right" and "in/out" directly from the camera's own forward/right vectors at the moment the preset is applied (`right = cross(forward, world-up)`) rather than from Euler yaw sign, so they're geometrically unambiguous — verified numerically (see git history) rather than just reasoned through. Pan/Orbit presets are framed as "arrive at my current vantage from the named side" rather than "leave from here": `PRESET_OPPOSITE` maps each to its mirror (`pan_left`↔`pan_right`, `orbit_left`↔`orbit_right`); applying e.g. Pan Left computes the START pose by running `pan_right`'s transform on the current pose (camera's initial gaze is offset to the right of the geo, since a true pan never moves position) and sets the END to the current pose unchanged, so the move always finishes on whatever you're currently framing. Orbit is the same computation but the offset is positional (start position literally sits to the right of the pivot for Orbit Left) rather than angular, since orbit moves the camera itself. Dolly has no left/right, so it keeps the simpler current-pose-is-the-start shape. Applying any preset replaces the current keyframe list (2 keyframes, `frame_index` 0 and `frame_count-1`, `ease_in_out`) rather than appending, since presets are a fresh quick-start. **FBX camera import** ("📥 Import Camera FBX" in the same panel) reads a DCC-authored camera move (Blender/Maya export) via Three.js's `FBXLoader`/`AnimationMixer`, client-side only (never touching `atlas_camera.core`, same rule `OBJLoader` already follows) — parses the uploaded file, finds the first `THREE.Camera` in the FBX with an `AnimationClip`, and samples its world transform at N evenly-spaced times across the clip's duration (`N` = the "samples" input, default 30). An FBX export has no ground-truth relationship to Atlas's world frame — the same problem `AtlasAddPatchView` solves for one static offset via a constructed (not solved) patch camera — so this treats the whole clip as a **relative** move applied on top of wherever the viewport camera currently is: `alignQuat = Quaternion.setFromUnitVectors(fbxForward_0, baseForward)` aligns the FBX camera's own frame-0 forward to the current base forward (`captureCurrentPose()`, shared with "+ Keyframe" and the presets), then every sample's position-delta-from-frame-0 and forward vector are rotated by that same `alignQuat` before being added onto the base pose — verified numerically for pure translation (exact) and pure level-pan (exact angle transfer). A "scale" input (default 1.0) covers the common FBX-cm-vs-solved-meters unit mismatch, adjusted by eye like `flip_azimuth`. **Known limitation:** `alignQuat`'s rotation axis only commutes with the FBX clip's own rotation when both cameras are reasonably level (same assumption `horizon_row_from_extrinsics` already makes) — a level FBX pan transfers its exact angle regardless of the two cameras' absolute yaw offset, but a steeply pitched FBX camera aligned to a very differently-pitched base view can pick up rotational "swim" beyond a pure transfer; acceptable for a first pass, recalibrate by eye if it looks off. Imported samples land in `pathFrameCount`/`pathFps` directly (overwriting them to match the import, unlike presets which respect the existing `frame_count` widget) with `easing:"linear"` throughout, then flow through the exact same Play/Bake/`AtlasExportCameraPathUSD` pipeline as hand-authored or preset paths.
- **Detached viewport controls (`AtlasViewportControls` + `AtlasBlockoutViewport`'s `controls` input):** by the time Camera Path (presets + FBX import) landed on top of the primitive toolbar, Project/Diagram/Info toggles, and Render Passes, `AtlasBlockoutViewport` had accumulated enough buttons/panels to crowd out the actual 3D view. `AtlasViewportControls` is a companion node with no computation of its own — its single output (`ATLAS_VIEWPORT_LINK`) carries no real data; the graph LINK exists purely so each node's frontend JS can find the OTHER node's live instance (`getLinkedControlsNode`/`getLinkedViewportNodes` in `atlas_blockout.js`, walking `node.graph.links` rather than `app.graph` so this keeps working inside subgraphs). When linked, the viewport's `buildNodeUI` reparents its ALREADY-BUILT `toolbar`/`pathPanel` DOM elements (`mountControls()`) into the Controls node's container instead of its own — every button's `onclick` closure still references the exact same `scene`/`camera`/`fly`/`pathKeyframes` variables it always did, so this is a pure DOM relocation, not a rewritten API surface, and carries zero behavior-change risk. `mountControls()` re-runs on `onConnectionsChange` (either node) and once more when the Controls node's own container is created, covering both load-order races in a saved workflow. **Backward compatible by construction:** with nothing connected to `controls`, `mountControls()` falls back to the viewport's own container in the exact original DOM order (toolbar, pathPanel, canvas) — so every workflow saved before this split keeps working unchanged. `mountControls()` also short-circuits (`_atlasMountTarget`) if called again with the same target, so repeated `onConnectionsChange` firings never re-append/thrash the DOM. **Resize history — one failed attempt, one working fix, both instructive:** a `node.onResize` handler that called `resizeViewport(w, h)` (changing `canvas.width`/`canvas.height`) to re-render at the dragged size froze the entire tab. Root cause, confirmed by reading the real installed `scripts/domWidget.ts` (extracted from its sourcemap, since it's only documented as "internal, may change"): ComfyUI's own DOM-widget layout math (`BaseDOMWidgetImpl.computeLayoutSize`, upstreamed into litegraph's own node resize handling — `addDOMWidget` already chains its OWN `node.onResize` via `useChainCallback`) sizes the widget from `getMinHeight`/`getMaxHeight`/`getHeight` options (defaulting to a bare 50px minimum if none are given) — nothing here should itself loop. The actual feedback path was CSS: the canvas's `height:auto` meant changing its intrinsic `width`/`height` attributes changed its *laid-out* height too, and neither `container` nor `canvasWrap` had an explicit `height:100%` chain down to the canvas, so dragging the node bigger gave the DOM widget more space that nothing in the chain actually consumed — the widget's real rendered height stayed tied to the canvas's own intrinsic aspect, which is what "resizing resets to a small version" was: not a bug in a resize *handler*, but a layout chain that never grew in the first place. **Fix:** `container` (the addDOMWidget element) now has `height:100%`; `canvasWrap` is `flex:1;min-height:0` (a flex child of `container`, which is `display:flex;flex-direction:column`); the canvas itself is `height:100%` (not `auto`) of `canvasWrap`. `addDOMWidget` also now passes `getMinHeight() { return 240 }` (the sanctioned option) instead of leaving LiteGraph's fallback-50px floor unset. Critically, **no JS resize hook is involved at all** — the canvas's own `width`/`height` attributes (intrinsic WebGL buffer / render resolution, still governed solely by the `resolution` widget) are never touched by resizing; the browser just rescales the existing buffer via ordinary CSS layout, which cannot feed back into anything. `resolution` remains the only way to change actual render resolution; dragging the node just changes displayed size (may blur if dragged far past the current resolution). **Follow-on bug from that same fix:** giving `canvasWrap`/`container` real flex sizing (`flex:1`, `height:100%`) surfaced a second, well-known CSS gotcha — flex items default to `min-width:auto`, which resolves to their content's min-content size, and for a `<canvas>` (a "replaced element" like `<img>`/`<video>`) that min-content floor is its INTRINSIC width (the `width` *attribute*, e.g. 768px — `width:100%` in CSS only affects the *used* size, never this floor). That silently forced `canvasWrap`, and the node containing it, to never shrink below the canvas's intrinsic pixel width regardless of the node's actual dragged size, surfacing as the node snapping/stretching wider on the first interaction that triggered a relayout (mousedown to orbit). Fixed with the standard remedy: explicit `min-width:0` on both `canvasWrap` and `container`, overriding the `auto` default. **Third follow-on, found via ShotCam testing:** `width:100%;height:100%` on the canvas has no aspect-ratio awareness — it always stretches the WebGL buffer to exactly fill its box, which was invisible before because the box's shape and the render resolution's aspect were always coincidentally close (no prior workflow set a render resolution far from the node's on-screen shape). `AtlasDefineShotCam`'s `shot_cam` (see "ShotCam" below) can conform the render resolution to a very different aspect (e.g. 2.39:1 anamorphic) than whatever shape the node happens to be on screen, making the stretch/squash obvious. Fixed with `object-fit:contain` on the canvas (plus a matching `background:#111` on `canvasWrap` so the resulting letterbox bars read as intentional, not a gap) — pure CSS, no new JS resize logic, verified live via `getBoundingClientRect()`/`getComputedStyle` before/after. **Known limitation, not fixed in this pass:** the diagram (📊)/HUD (ℹ) SVG overlays are still sized to the full `canvasWrap` box (`inset:0`,`width/height:100%`), so they'll misalign with the now-letterboxed canvas content whenever a significant aspect mismatch triggers letterboxing — narrow (only visible with those toggles on AND a shot cam far from the node's on-screen aspect), left for a follow-up rather than risking a flexbox+`aspect-ratio` rewrite in a spot with three prior documented bugs.
- **`scene_type` convenience preset, now the primary control (still artist-selected, not auto-detected):** `AtlasDeriveProjectionGeometry`'s `scene_type` widget (default `manual`) grew from 4 to 9 options — `organic`, `mountains`, `forests`, `aerial`, `indoor`, `outdoor`, `simple_walls`, `towers_spires` — specifically to close the "primitive_method is silently ignored when geometry_mode=relief_mesh" confusion: each option is a complete, self-consistent bundle of `geometry_mode`/`primitive_method`/`relief_quality`/`depth_edge_rel`/`max_objects`/`depth_model`, resolved in a class-level `_SCENE_TYPE_PRESETS` dict at the top of `derive()` — purely a convenience mapping onto the exact same code paths, never a new solving strategy. `mountains`/`forests` are both `relief_mesh` at `relief_quality="high"` (terrain/canopy need more grid density than the default to read as continuous); `forests` additionally relaxes `depth_edge_rel` to `1.0` since dense canopy depth is genuinely noisy at small scale and the default threshold shreds it into holes (see the relief-mesh-tearing entry above — this is that same finding, applied). `aerial` is `geometry_mode="both"` + `azimuth_walls` + `max_objects=6` — buildings as boxes over a relief-mesh ground/treeline, the drone/top-down case. `simple_walls`/`towers_spires` newly expose `azimuth_walls`/`vertical_extrusion` as named presets (previously only reachable by manually setting `geometry_mode=primitives` — `scene_type` couldn't reach them at all before). `manual` (default) leaves every widget exactly as set — fully backward-compatible; all-new option VALUES were *added*, none renamed, so workflows saved before this expansion keep working unchanged.
  **Widget visibility follows the preset** (`atlas_camera/comfy/web/atlas_derive_geometry.js`, extension `AtlasCamera.DeriveGeometryPresets`, independent of `atlas_blockout.js`): picking anything but `manual` hides whichever widgets that preset has made moot, via `computeHiddenWidgets()` — always `geometry_mode`/`primitive_method` for any non-manual preset, plus that preset's own explicit override keys, plus `relief_grid`/`relief_quality`/`depth_edge_rel` whenever the resolved `geometry_mode` is `primitives` (no relief mesh ever builds) or `relief_quality` is explicitly set (it always supersedes `relief_grid`), plus `max_objects`/`max_walls` whenever `geometry_mode` is `relief_mesh`-only (objects/walls still get derived internally for `ground_scale`/backdrop, per the design rule above, but are discarded from the kept geometry). A widget only disappears when changing it would have *zero effect* — everything left visible still does something. `SCENE_TYPE_PRESETS` is hand-mirrored in that JS file (same accepted-duplication pattern as `camera_path.py`/`catmullRom3JS` — keep both in sync by hand) purely to decide what to hide; Python's `_SCENE_TYPE_PRESETS` remains the only place the actual override logic runs. Hiding uses the standard litegraph trick — swap `widget.type` to an unrecognized string and `computeSize` to `[0,-4]`, restoring both on re-show — never touches `.value`, so serialization/queueing behave exactly as if the widget were visible.
  The preset only reaches this node's own depth estimation; if `AtlasLearnedSolveFromImage` upstream uses `height_mode=measure_from_depth`, its separate `depth_model` widget must be set to match by hand (nodes don't cross-configure each other's widgets). Recorded in `debug_metadata["proxy_derivation"]` (`scene_type`, `depth_edge_rel`, `relief_grid`, `relief_quality`, `max_objects`) for traceability.
- **Composable geometry derivation (`AtlasDepthMap` + `AtlasDeriveReliefMesh`/`Walls`/`TowersSpires`/`RoofsFacades`/`InteriorRoom` + `AtlasMergeGeometry`):** the Nuke-style alternative to `AtlasDeriveProjectionGeometry`'s `scene_type` presets, for scenes that genuinely mix strategies (foreground buildings over background terrain, say) rather than fitting one named preset. Instead of one node with a combo that locks/hides other widgets, each derivation strategy is its own node with *only* parameters that always apply — no `geometry_mode`/`primitive_method` cross-product exists to be confusing, so none of these nodes need `atlas_derive_geometry.js`'s widget-hiding trick. `AtlasDeriveProjectionGeometry` itself is untouched — this is additive, not a replacement, and the two private helpers these new nodes share (`_solve_camera_params`, `_horizon_y_from_solve`, `_depth_map_for_solve`, `_replace_proxy_role_geometry` in `nodes.py`) intentionally are **not** used to refactor the existing node, to avoid risking regression in tested, shipped behavior.
  **Depth is shared, not just for speed.** `AtlasDepthMap` runs Depth-Anything once and passes the raw `DepthResult` by reference as a new `ATLAS_DEPTH_MAP` type (same zero-serialization custom-type pattern as `ATLAS_SOLVE`/`ATLAS_CAMERA_PATH`/`ATLAS_VIEWPORT_LINK` — an arbitrary string type name, Python object passed in-process, no JSON round-trip). Every extraction strategy (`build_relief_mesh`, `derive_projection_proxies`, `derive_vertical_extrusion_proxies`, `extract_planes_ransac`, `extract_room_cuboid`) shares one contract — 2D float64 `(H,W)` forward-Z array + `view_matrix`/`fx,fy,cx,cy` — confirmed identical across all five call sites, so one `AtlasDepthMap` output safely fans out to several derive nodes. This is required for **correctness**, not just efficiency: each strategy independently fits its own ground plane from whatever depth map it receives, so two branches estimating depth separately could disagree slightly on metric scale, landing their ground planes at different heights and misaligning the merged result. Distinct from `AtlasDepthAnything` (existing): that node's IMAGE output is a lossy, per-image min-max-normalized preview — the real near/far distances and `is_metric` flag are computed then irrecoverably discarded — so it cannot be repurposed as the metric carrier.
  **Chaining derive nodes silently loses data — this is why the merge node exists.** Every derive node (new and the original `AtlasDeriveProjectionGeometry`) deep-copies its input solve and then strips *all* prior `PROXY_ROLE`-tagged geometry before adding its own (`_replace_proxy_role_geometry`) — deliberate, so re-running a derive node never piles up stale geometry across repeated executions. The side effect: wiring `AtlasDeriveWalls`'s output solve directly into `AtlasDeriveReliefMesh`'s `solve` input does **not** accumulate both branches' geometry — the second node erases the first's. `AtlasMergeGeometry` is the one explicit, visible place two branches' `proxy_geometry` lists actually combine — binary, matching Nuke's own Merge node, chainable for 3+-way combination rather than a fixed N-ary input cap. It takes `solve_a`'s full deep copy as the base and appends **only `solve_b`'s `PROXY_ROLE`-tagged geometry** (what `solve_b`'s own derive node actually added), never `solve_b`'s full `proxy_geometry` list. That distinction was found via live end-to-end verification, not reasoned out upfront: every solve used to inherit a placeholder `"ground_plane"` entry (`role="ground"`, a *different* role, not `PROXY_ROLE` — see the "two other unrelated `role`-key conventions" note that governs `proxy_geometry.py`'s own filtering) from `projection_scene.create_default_projection_scene()`, and every derive node passed that pre-existing entry through untouched — so naively appending `solve_b`'s entire list duplicated it on top of `solve_a`'s own copy of the exact same thing, even though `solve_a` already provided it. Filtering `solve_b`'s contribution to `role == PROXY_ROLE` fixes this in one place for any current or future non-`PROXY_ROLE` pass-through geometry, not just this one case — kept even after the specific placeholder below was removed, since it's the correct general contract regardless. It also **dedupes `"projection_backdrop"`** within that same `PROXY_ROLE` set: every extraction strategy always emits exactly one primitive with that literal name (a far catch-all plane), and `atlas_blockout.js` has a name-based special case for it — merging two branches that each derived one would still produce two overlapping backdrop planes without this, so the merge keeps only the first one encountered (`solve_a`'s). No viewport-side changes were needed otherwise: `buildDerivedProxies` already treats `proxy_geometry` as one flat, source-agnostic list. Verified live against a real photo (`AtlasLearnedSolveFromImage` → `AtlasDepthMap` → `AtlasDeriveWalls` + `AtlasDeriveReliefMesh` → `AtlasMergeGeometry` → `AtlasExportSolveJSON`, comparing the merged JSON against each branch exported alone) and then smoke-tested across 5 more varied photos: merged geometry always carried exactly one `ground_plane`/one `projection_backdrop` (never duplicated), each branch's own contribution intact (walls/boxes/cylinders varying sensibly per photo, relief mesh always present), no NaN/Infinity in any output.
- **The "ground" naming confusion, resolved by removing the actual dead weight (`projection_scene.py`):** the live verification above surfaced a second, more interesting issue than a naming collision — `create_default_projection_scene()` (called by several `solver.py` solve-construction paths) used to seed every solve with a placeholder `"ground_plane"` primitive (`role="ground"`, a 10×10m flat default). Exhaustive grep across the whole repo (Python, the React `ui/`, JS) found **zero consumers**: `serialize_proxy_geometry` only ever sends `role=="projection_proxy"` (`PROXY_ROLE`) primitives to the viewport, so it was never rendered; no exporter reads it; `up_axis`/`coordinate_system` are already their own fields on `AtlasProjectionScene`, so even its metadata was redundant. `proxy_box`/`add_axis_guides` in the same module are equally unused outside their own test file (left alone — out of scope for this specific fix, since they're unused-but-harmless rather than actively confusing). Its one real, observed effect was confusion: an almost-identically-named, *actually rendered* `"projection_ground"` primitive (`PROXY_ROLE`, from `derive_projection_proxies`/`AtlasDeriveWalls`) sits in the same list, and the placeholder was the root cause of the `AtlasMergeGeometry` duplication bug above. Fixed by deleting the placeholder from `create_default_projection_scene()` entirely — rather than renaming it — since nothing needs it; a solve's `proxy_geometry` now starts genuinely empty. If a real default-ground visualization is ever wanted again, it should be built as a proper `PROXY_ROLE` primitive by whichever node actually wants to show it, not injected invisibly at solve-construction time.
  **Known, documented risk, not fixed in this pass:** `derive_projection_proxies` (azimuth_walls) carries its own inline copy of the ground-fit/scale math rather than calling the shared `depth_geometry.fit_ground_and_scale` that `derive_vertical_extrusion_proxies`/`extract_planes_ransac`/`extract_room_cuboid` all call — confirmed line-for-line identical arithmetic today (bit-identical results on identical inputs), but it's two independently-maintained copies that happen to agree, not one shared function. Sharing one `ATLAS_DEPTH_MAP` across branches removes the "different depth estimate per branch" risk entirely; if these two copies ever drift apart in a future edit, cross-strategy merges could silently misalign. Pre-existing, out of scope for this feature.
- **ShotCam — a project-level render/output camera format (`AtlasShotCam` + `AtlasDefineShotCam` + `AtlasMergeGeometry`'s/`AtlasBlockoutViewport`'s `shot_cam` input):** every `AtlasSolve` carries its own `AtlasIntrinsics` derived from whatever photo was actually solved — its own resolution, aspect ratio, guessed sensor/lens. There was no concept above a single per-image solve, which becomes a real gap once a graph combines geometry from more than one source image (`AtlasMergeGeometry`'s own docstring already admitted this: *"does not check for or correct a mismatch between solve_a and solve_b"*), and `AtlasBlockoutViewport` always derived its render resolution/aspect from whatever `source_image` happened to be wired in. `AtlasShotCam` (schema.py) is a small, JSON-safe, intrinsics-only dataclass — `sensor_width_mm`/`sensor_height_mm`/`focal_length_mm`/`resolution_long_edge_px` — analogous to a Nuke/Resolve project format setting: it describes the FINAL render/export camera, decoupled from any individual solved photo's own sensor/lens/aspect. Deliberately has no position — camera placement always still comes from whichever solve's own recovered extrinsics is already in play. `atlas_camera.core.intrinsics.intrinsics_from_shot_cam()` turns one into real pinhole `AtlasIntrinsics` via the existing `build_intrinsics()`/`focal_length_mm_to_pixels()` (no new pinhole math). `AtlasDefineShotCam` is a pure config-carrier node (same spirit as `AtlasViewportControls`), output type `ATLAS_SHOT_CAM` (same zero-serialization in-process pattern as `ATLAS_SOLVE`/`ATLAS_DEPTH_MAP`).
  **Two entry points, one precedence rule.** `AtlasMergeGeometry`'s optional `shot_cam` input, when connected, is a pure *attachment* — `out.shot_cam = shot_cam` — never a mutation of `out.camera` (which stays `solve_a`'s own intrinsics/extrinsics exactly as before). This lets a project format defined once flow downstream to whatever consumes it, without rewiring. `AtlasBlockoutViewport` also takes an optional direct `shot_cam` input; its precedence is resolved once at the top of `render()`: an explicitly-wired `shot_cam` wins, else `solve.shot_cam` (inherited from an earlier merge) is used if present, else today's `_fit_long_edge(source_image aspect)` behavior — fully backward compatible when neither is present. Verified live end-to-end via both paths against real solved data: a direct wire and a merge-inherited attachment both correctly conformed independent `AtlasBlockoutViewport` instances to the identical shot format.
  **Why this is safe — and the one place the naive implementation would have broken things:** geometry is world-space and metric; it doesn't care about sensor/lens format, so ShotCam never needs to touch any derive node. The one real hazard, found and fixed during implementation (not anticipated in the original design): `_extract_blockout_camera`'s top-level `fx`/`fy`/`cx`/`cy`/`image_width`/`image_height` are read by TWO different frontend consumers from the exact same payload object — `applyRecoveredCamera` (the live orbit/viewing camera's FOV/aspect) AND `makeProjectionMaterial` for the PRIMARY source's own texture-sampling uniforms (`applyCamera(data)` and `setProxies(data)` in `atlas_blockout.js` are both called with the identical `data` object from the same execution). Naively overwriting `fx`/`fy`/`cx`/`cy` to conform to a shot cam would have silently corrupted how the actual photo gets projected onto geometry. The fix: two new, genuinely separate payload keys, `render_fy`/`render_image_height`, read ONLY by `applyRecoveredCamera` (`atlas_blockout.js`, one small edit — `data.render_fy ?? data.fy`, `data.render_image_height ?? data.image_height`, defaulting to today's values when no shot_cam is present) — `fx`/`fy`/`cx`/`cy`/`image_width`/`image_height` themselves are never touched, so `makeProjectionMaterial` for the primary (and every patch's own `ProjectionSource` camera, entirely unrelated to any of this) keeps sampling correctly regardless of what shot format is active. `target_width`/`target_height`, by contrast, needed no separate key at all — they're already routinely decoupled from `image_width`/`image_height` today (`_fit_long_edge` resizes the canvas to `resolution` regardless of the source photo's actual pixel dimensions), and `makeProjectionMaterial` never references them, confirmed by reading its exact uniform list. `camera_meta` (the ℹ Info/📊 Diagram diagnostics — `focal_mm`, `fov_h_deg`, etc.) is also left untouched by a shot cam, since those describe how the photo was actually shot, not the output format.
  **Deferred, not in this pass:** none of `AtlasExportUSD`/`AtlasExportReliefMesh`/`AtlasExportMayaReviewScene` currently has any resolution/format override — camera params come straight from `solve.camera`/`intr.image_width`/`intr.image_height` in all three. Extending the same `shot_cam`-input pattern to these exporters is a structurally identical follow-on, deliberately out of scope here to keep the first pass small and reviewable.
- **Relief mesh tearing on noisy AI-image depth (`relief_grid`/`depth_edge_rel`/`relief_quality`):** `build_relief_mesh` tears a quad into a hole whenever its corner depths differ by more than `depth_edge_rel` (relative) — this is *load-bearing* (silhouettes must not rubber-sheet onto the background), but a genuinely noisy monocular depth estimate (common on complex/reflective/low-light real photos) can trip that threshold across large swaths of the mesh even where there's no real discontinuity, producing a visually "shattered"/shard-like result rather than a continuous surface — confirmed empirically (not just reasoned through): on a synthetic depth map with exactly one real edge, `torn_fraction` drops monotonically as `grid_long_edge` rises (32→256 took it from 0.040 to 0.006), because a finer grid means each quad spans less real-world area and is less likely to straddle spurious per-pixel noise. `AtlasDeriveProjectionGeometry`'s `relief_grid` (default **128**, range 16–1024 — was 96/256 before this was diagnosed; now matches `AtlasExportReliefMesh`'s own default) and new `depth_edge_rel` widget (default 0.5, same param/default as `AtlasExportReliefMesh` — previously hardcoded, not exposed on this node at all) are both wired straight into that same `build_relief_mesh` call. `relief_quality` (`custom`/`low`/`medium`/`high`/`ultra`, default `custom`) is a `relief_grid` quick-pick — `low=64, medium=256, high=512, ultra=1024` — following the exact same convenience-preset pattern as `scene_type` above (a class-level `_RELIEF_QUALITY_PRESETS` dict resolved at the top of `derive()`, `custom` leaves `relief_grid` exactly as its own widget says, fully backward compatible). `ultra` trades a much heavier mesh (bigger solve JSON, slower viewport) for the fewest holes — recommend trying `high` first on troublesome photos. Both `relief_grid` and `relief_quality` are recorded in `debug_metadata["proxy_derivation"]` for traceability.
- Output dimensions **auto-adopt the source image** — never hardcoded. The solve carries `camera.intrinsics.image_width/height`; `AtlasGroundDepthMap`/`GroundMask`/`HorizonMask` take `image_width`/`image_height` where **0 = auto** (via `_solve_image_size`), and `AtlasBlockoutViewport` takes a single `resolution` (long edge) and derives W×H from the source aspect (via `_fit_long_edge`). The blockout frontend resizes its canvas/camera to those `target_width/target_height` on execution, so the viewport inherits the image aspect.
- Camera height / metric scale is **measured, not assumed**, via a tiered cascade (best evidence first), each adopted only above its confidence threshold and otherwise surfaced as a flagged candidate (never silently promoted — matches the LLM-suggestion confirm principle):
  1. **Reference object** (`scale_references` / `AtlasReferenceScaleSolve`) — known-size object (person/door/car from `reference_data`, or explicit height) solved by single-view geometry. Most reliable; `_REFERENCE_ADOPT_CONFIDENCE`. VLM cues auto-feed this tier: `AtlasVLMScaleCues` → `scale_references` JSON → `AtlasApplyScaleReferences` (which rescales **only on `confirm`** — `multimodal_helper.scale_references_from_observation()` maps `SceneScaleCue`s to specs via the registry; LLM cues are never auto-promoted).
  2. **Depth ground-plane** (`camera_height="auto"`) — Depth Anything V2 + `estimate_ground_height_from_depth`; `_HEIGHT_ADOPT_CONFIDENCE`. Unreliable on AI imagery, so usually low-confidence.
  3. **Assumed default** — last resort, flagged. `solve.debug_metadata["scale_source"]` records which tier won.
- The Three.js 3D viewport is a frontend dependency only; it must not become a dependency of `atlas_camera.core`.
- `AtlasDecomposeSolve` is the correct place to expose `horizon_angle_deg` (reads from `solve.debug_metadata`). `AtlasDecomposeCamera` does NOT expose it (camera object has no horizon data).

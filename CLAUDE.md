## Commands

```powershell

pip install -e ".[dev]"

# Run all tests
python -m pytest -q

# Run a single test file
python -m pytest tests/test_intrinsics.py -v

# Run a specific test
python -m pytest tests/test_intrinsics.py::test_focal_length_to_pixels -v

# Install with UI and vision support
pip install -e ".[ui,vision]"

# Install neural extras (GeoCalib learned prior + Depth Anything V2) ŌĆö needed

# ComfyUI's venv); GeoCalib is GitHub-only and not on PyPI.
pip install -e ".[neural]"
pip install "git+https://github.com/cvg/GeoCalib.git"

# Start the FastAPI backend
python -m atlas_camera.ui

# Start the React frontend (separate terminal)
cd ui && npm install && npm run dev

# Install atlas_camera into ComfyUI's venv (editable ŌĆö run once)
# Replace <COMFYUI_ROOT> with your local ComfyUI install path.

# Verify ComfyUI can import the package
& "<COMFYUI_ROOT>\venv\Scripts\python.exe" -c "import atlas_camera; print(atlas_camera.__file__)"
```

## Optional dependency groups

- `dev` ŌĆö numpy, opencv-python, pytest
- `vision` ŌĆö numpy, opencv-python (runtime vision)
- `image` ŌĆö Pillow only
- `usd` ŌĆö usd-core
- `ui` ŌĆö FastAPI, uvicorn, Pillow, python-multipart
- `neural` ŌĆö torch + GeoCalib (learned single-image camera prior). GeoCalib is GitHub-only: `pip install "git+https://github.com/cvg/GeoCalib.git"`. torch is expected from the host env (e.g. ComfyUI's venv).

The core package has **zero required runtime dependencies**. All vision, USD, and UI imports are guarded with informative `ImportError` messages.

## ComfyUI integration (`atlas_camera/comfy/`)

### Setup

A symlink connects the node pack to ComfyUI's custom_nodes directory (paths below
are this machine's; substitute your own ComfyUI install and repo checkout locations):

```
<COMFYUI_ROOT>\custom_nodes\AtlasCamera
    ŌåÆ <REPO_ROOT>\atlas_camera\comfy
```

The package is installed in editable mode in ComfyUI's venv so `import atlas_camera` resolves to this project directory. Changes to Python source are live immediately ŌĆö no reinstall needed.

### Clone-and-go entry point (repo root `__init__.py`, 2026-07-11)

The repository root is itself a loadable ComfyUI custom node: `git clone` into
`custom_nodes/` works with no pip install (the root `__init__.py` prepends the
checkout to sys.path, re-exports the mappings from `atlas_camera.comfy`, and
sets a RELATIVE `WEB_DIRECTORY = "./atlas_camera/comfy/web"`). The dev
symlink+editable setup never loads that file. `atlas_camera/comfy/__init__.py`'s
own `WEB_DIRECTORY` is now the conventional relative `"./web"` (was absolute ŌĆö
worked only because os.path.join discards the left side for absolute right
sides). pyproject carries `[tool.comfy]` (PublisherId `miikejamesburns`) for
`comfy node publish`; the publisher must exist on registry.comfy.org first.
Pinned by `tests/test_node_pack_entrypoint.py`, which loads the root file the
way ComfyUI would.

### Double-import guard (critical)

`atlas_camera/comfy/__init__.py` is loaded twice at ComfyUI startup:
1. As `AtlasCamera` custom node (by ComfyUI's node loader)
2. As `atlas_camera.comfy` package (by `from atlas_camera.comfy.nodes import ...` inside the same file)

Both loads hit the same file, causing the aiohttp route `GET /atlas/camera_data/{node_id}` to be registered twice, which raises `RuntimeError: method HEAD is already registered`. The fix in `__init__.py` checks `if not any(r.path == ... for r in _routes)` before registering.

**Projection shader was missing its linearŌåÆsRGB output encode (fixed 2026-07-06):** discovered while investigating why ­¤ōĮ Project looked artificially dark/desaturated relative to the source photo ŌĆö the same symptom that motivated adding movable lights above, but a distinct and more fundamental bug underneath it. `renderer.outputColorSpace = SRGBColorSpace` and every source texture is tagged `colorSpace = SRGBColorSpace`, so `texture2D(uTexture, uv)` in `PROJECTION_FRAGMENT_SHADER` already returns correctly-**decoded linear** values (the GPU auto-decodes an sRGB-tagged texture's bytes to linear float on sample, transparently, regardless of which material/shader does the sampling). But `PROJECTION_FRAGMENT_SHADER` is a raw `THREE.ShaderMaterial` ŌĆö Three.js only auto-appends its own linearŌåÆdisplay-colorspace re-encode chunk (`colorspace_fragment`) into *built-in* material shader templates (`MeshStandardMaterial`, `MeshBasicMaterial`, etc.); a fully custom `ShaderMaterial` never gets it unless the shader author writes the equivalent GLSL by hand. So the shader was writing correctly-decoded linear values straight to `gl_FragColor` with no re-encode ŌĆö every projected pixel silently skipped the gamma step the display expects, reading as darker/flatter than the source photo (independent of, and in addition to, whatever the ŌśĆ Exposure/­¤ÆĪ Lights sliders were set to). Fixed by adding `atlasLinearToSRGB()` (matches THREE.ShaderChunk's own `LinearTosRGB` formula, exponent `0.41666 Ōēł 1/2.4`, verified numerically against the exact sRGB OETF) and applying it to `col.rgb * relight` (clamped to `[0,1]` first, since the encode's `pow()` is undefined for negative/NaN-prone inputs and the relight term can push values above 1) right before the final `gl_FragColor` assignment. This is orthogonal to the ŌśĆ Exposure bullet above, which is still accurate as written ŌĆö exposure/tonemapping remain genuinely absent from this shader; only the separate color-*space* encode step was missing. Verified live: captured the exact same frame with and without the encode (temporarily patching `material.fragmentShader` + `needsUpdate=true` to compare) ŌĆö the un-encoded version was visibly crushed/muddy, the encoded version matched the source photo's contrast and saturation.

**Orbit coverage (why the mesh/projection can look like it "disappears" on rotate):** derived geometry only ever covers what the recovered camera could see ŌĆö a forward-facing cone, inherent to single-photo reconstruction. `createOrbitControls` clamps yaw/pitch to ┬▒80┬░/┬▒55┬░ around the recovered camera's own direction (`theta0`/`phi0`, wraparound-safe via `atan2(sin,cos)`, re-anchored on every `syncFromCamera()` ŌĆö i.e. every execution and every ­¤ōĘ Camera View click) so you can't orbit past the reconstructed cone into empty space in the grey/undressed preview.

**Orbit pivot ŌĆö geometry centroid, not a ground heuristic:** `applyRecoveredView` sets an initial pivot via `groundPointInView` (where the camera's forward ray crosses Y=0, capped at the solved scene depth) purely as a fallback for before any geometry exists (or for workflows that never run `AtlasDeriveProjectionGeometry` at all) ŌĆö it's a generic heuristic, not the actual centre of whatever gets reconstructed, and could be visibly off for reconstructions that aren't roughly centred on the camera's exact forward ray. Once `buildDerivedProxies` has real geometry, `setProxies` immediately overrides it with `computeGeometryPivot(data)` ŌĆö since 2026-07-09 the **recovered camera's central view ray at the MEDIAN sampled vertex depth** of the `atlas_derived_proxies` meshes (Ōēż800 samples/mesh, recovered pose from the payload's `view_matrix` so the pivot never depends on the user's current orbit), still excluding `"projection_backdrop"`; patch/clean-plate sources live in their own `atlas_patch_N` groups and are never included. It was previously a `THREE.Box3` bounding-box CENTER, which is `(min+max)/2` and therefore tail-dominated ŌĆö fine for clustered fitted primitives, but on a single full-scene relief mesh (the hidden-geometry workflows' base geometry, spanning near-foreground to far-clip plus fill/outpaint skirts) it parked the pivot deep behind the subject (artist-reported). Median vertex depth Ōēł the depth of the middle of the visible surface area, since relief grids sample the image uniformly. This only calls `controls.setTarget(...)` + `syncFromCamera()` (re-deriving the orbit sphere from wherever the camera *already* is) ŌĆö it never moves the camera itself, so recomputing it on every execution (including re-executions from ŌÅ║ Bake Path) can't disrupt an in-progress inspection, and is a no-op in effect when the geometry hasn't changed between runs.

`AtlasBlockoutViewport`'s `preview_expand` widget (default 1.0 = off) can optionally dilate the geometry itself for more coverage within that arc, via `proxy_geometry.dilate_proxy_geometry_for_preview()`: for any point with local normal n╠é, radiating from the camera position, `p' = pivot + ((p-pivot)┬Ęn╠é)n╠é + scale┬Ę[(p-pivot) - ((p-pivot)┬Ęn╠é)n╠é]` ŌĆö only the normal-perpendicular offset scales, so a plane widens without drifting in depth, a box/cylinder (no single normal) dilates uniformly, and the relief mesh dilates per-vertex using each vertex's own (genuinely arbitrary) normal. **Known interaction, not further fixable without dual geometry (dilation vs. ­¤ōĮ Project):** dilation and the projection shader are fundamentally in tension, and this is *not* the same bug the clamp fixes. The ­¤ōĮ Project shader assigns texels by casting each fragment's world position through the RECOVERED (undilated) camera and discarding anything that lands outside the actual photographed frame (`atlas_blockout.js`'s `PROJECTION_FRAGMENT_SHADER`, `discard` on out-of-bounds/behind-camera). Dilated geometry is, by construction, surface area the camera never actually photographed ŌĆö there is no real pixel data for it, so it correctly discards to empty/black. Because dilation is most visible away from dead-center, this shows up as large black regions after only a *moderate* orbit with Project active ŌĆö confirmed via live reproduction: with the prior default (`preview_expand=1.4`), a 60px drag left roughly half the frame black; the identical drag at `preview_expand=1.0` showed full coverage (parallax-shifted, as expected). This is why the default was changed to 1.0 ŌĆö raise it only when inspecting undressed blockout primitives, never while relying on ­¤ōĮ Project.

## Coordinate conventions

- World: **right-handed, Y-up** by default
- **Recovered camera faces world ŌłÆZ** (canonicalized 2026-07-10 ŌĆö `solver._face_camera_toward_negative_z`, applied in BOTH solve paths): yaw is unobservable from a single image, so the facing is a free convention, and ŌłÆZ matches Maya/Nuke default cameras. Before this, every DCC import needed a manual ŌłÆ180┬░ Y rotation (found by a real Maya lineup). The flip is a world-side RotY(180) ŌĆö gravity/pitch untouched (LEFT-multiply on the cam_to_world block; the transposed side rotates in the CAMERA frame and inverts pitch ŌĆö guarded by tests)
- Image: **origin top-left, x right, y down**
- DCC conversions happen **only at adapter boundaries**, never in core
  - Blender (Z-up): converted in `atlas_camera/exporters/blender_exporter.py`
  - USD stage axis set at export time
  - OpenCV/NumPy: only imported in the `vision` optional layer

## Key design rules

- Core schema is pure Python dataclasses ŌĆö no external deps.
- Optional deps (`numpy`, `cv2`, `fastapi`, `pxr`) are always guarded by try/except with `pip install -e .[extra]` hints in the error message.
- Adapter boundaries must be explicit: coordinate-system conversions are never silent.
- `LatentComponent` slots (`depth`, `geometry`, `lighting`, `semantics`) default to empty until their solvers exist ŌĆö review packages describe unsupported components rather than omitting them. `depth` is now populated by `solve_still_image_learned(camera_height="auto")` (Depth Anything V2 map summary + measured camera height + confidence); `geometry`/`lighting`/`semantics` remain empty.
- **View-matrix convention (critical):** geometry/camera math must use the full 4├Ś4 `extrinsics.camera_view_matrix` end-to-end (`cam_to_world = inv(view_matrix)`, row-major, column-vector points, translation in column 3) ŌĆö the convention proven to match the viewport shader. Never build world math from the 3├Ś3 `camera_rotation_matrix` (it has a transpose ambiguity). `core/proxy_geometry.py`, `core/depth_geometry.py`, `core/plane_extraction.py`, `core/room_layout.py`, and `_ground_depth_compute` all follow this; `THREE.Matrix4.set()` consumes the row-major floats verbatim.
- **Multiple geometry-derivation strategies, always artist-selected, never auto-picked:** `AtlasDeriveProjectionGeometry`'s `geometry_mode` picks the output kind ŌĆö `relief_mesh` (default, a single triangulated mesh following actual depth contours ŌĆö handles arbitrary/organic shapes but has real gaps at sharp depth discontinuities), `primitives` (simple fitted shapes ŌĆö cleaner/blockout-like), or `both` (both at once, mostly for comparison; they visually overlap). When `primitives`/`both` is selected, `primitive_method` picks the fitting strategy: `azimuth_walls` (default ŌĆö vertical walls + foreground boxes/cylinders only, general-purpose but height is clipped to whatever 3D points individually pass a near-vertical-normal filter, so it truncates at a sloped roof/spire/tower ŌĆö confirmed on real church/tower photos), `ransac_planes` (any-orientation planes via sequential RANSAC seeded by a 2D orientation histogram ŌĆö best for exterior architecture: roofs, ramps, stepped facades), `room_cuboid` (Manhattan floor+Ōēż4 walls+optional ceiling aligned to the dominant wall direction ŌĆö best for genuinely orthogonal interiors; produces confidently wrong/skewed results on non-orthogonal rooms, so the artist must pick the right method for the shot rather than relying on auto-detection), `vertical_extrusion` (same wall orientation/distance detection as `azimuth_walls` via the shared `proxy_geometry._cluster_walls_by_azimuth` helper, but height comes from the image-space silhouette instead ŌĆö the topmost non-sky pixel per column, back-projected at its own depth regardless of local surface normal, per Hoiem/Efros/Hebert's "Automatic Photo Pop-up" (SIGGRAPH 2005) billboard-cutout technique; reaches towers/spires/sloped roofs `azimuth_walls` truncates, at the cost of representing them as a flat vertical plane rather than their true sloped shape). `core/depth_geometry.py` factors the shared back-projection/normals/ground-fit/backdrop logic (bit-for-bit consistent with `proxy_geometry.py`'s own copies, which are left untouched to avoid regression risk) so `proxy_geometry.py` (`azimuth_walls`, `vertical_extrusion`), `plane_extraction.py` (`ransac_planes`), and `room_layout.py` (`room_cuboid`) all agree on world points and metric scale for a given depth map. Every extractor's `stats` dict must include `ground_scale` ŌĆö required contract with the node's relief-mesh branch, which reuses whichever method's scale was computed. Cross-product normals aren't camera-oriented by construction; `plane_extraction.py`/`room_layout.py` flip them toward the camera globally right after back-projection (sign-invariant under the later ground rescale) so orientation metadata and per-side wall assignment are meaningful.
- **Foreground object height plausibility ceiling (`ProxyDerivationConfig.object_max_height_m`, default 6.0):** `_derive_objects`' foreground-box/cylinder clustering (shared by `azimuth_walls` and `vertical_extrusion` via the same `_derive_objects` call) groups candidate points into objects using a **2D (X,Z) occupancy grid** ŌĆö it has no notion of height when deciding what belongs to the same object. Found via live browser verification (not unit tests, which all used clean synthetic scenes): on a real photo, a genuine foreground object's footprint spuriously merged with an unrelated tall background structure behind/above it (one that wall-fitting hadn't already claimed) because they shared the same X/Z occupancy cells ŌĆö the resulting box's height (98th-percentile Y) reflected the tall structure, not the actual foreground object, producing an implausible ~14m "foreground" box that filled the entire viewport edge-to-edge. Mirrors the same "reject an implausible fit" pattern `_fit_cylinder` already uses for `cylinder_radius_range` (returns `None` outside the plausible radius band): when a cluster's fitted height exceeds `object_max_height_m`, refit using only the points at or below that ceiling (dropping the object entirely if too few remain) rather than passing an unbounded height through. Applies to both `derive_projection_proxies` and `derive_vertical_extrusion_proxies` since both call the same shared `_derive_objects`. Does not fix the underlying 2D-only clustering (a genuinely depth/height-aware clustering rewrite is out of scope here, and carries real regression risk for currently-working cases) ŌĆö this is a plausibility guard on the *output*, not a fix to the clustering algorithm itself.
- **Sky-aware depth (`depth_geometry.detect_sky_mask`):** monocular depth (Depth Anything and similar) has no reliable cue on feature-less sky/cloud regions and often hallucinates noisy, spatially-incoherent depth there rather than a smooth "far" value ŌĆö left unhandled, that noise gets triangulated into a jagged, spatially-huge mesh chunk (`relief_mesh.build_relief_mesh`) that dwarfs the actual photographed geometry, and can even leak into wall/primitive fitting. `detect_sky_mask(depth, horizon_y=...)` flags a pixel as sky when it's above the solved horizon line AND either at/beyond a far-depth percentile or has high local **roughness** ŌĆö mean-squared discrete Laplacian, not raw variance ŌĆö relative to the below-horizon baseline. Roughness (not variance) is deliberate: a genuinely sloped surface (a real roof, a ramp) has a constant local gradient, so its Laplacian is ~0 even though its raw variance in a window is large; plain variance would misclassify real sloped architecture as sky. `build_relief_mesh` uses this to exclude sky from triangulation entirely (a hole, like any other depth discontinuity) rather than merely distance-clamping it; `vertical_extrusion` (above) uses the same mask to find each wall's real silhouette top instead of guessing from noisy sky depth. **Self-disarms on interiors (`_sky_mask_incoherent`, 2026-07-13):** the roughness term is correct for genuine noisy sky but on an INTERIOR also fires on detailed ceilings / far greebled walls above the (arbitrary, sky-free) horizon, punching large scattered holes in real geometry (measured live: a sci-fi hangar lost 39% of its back wall; turning the heuristic off halved the missing fraction). Since the default can't flip globally (it's correct outdoors), `detect_sky_mask` now measures the candidate mask's SHAPE ŌĆö real outdoor sky is ONE region anchored to the top of frame and vertically contiguous; an interior misfire is scattered fragments with almost nothing top-anchored. Two numpy-only signals (`top_anchored` fraction + runs-per-column) separate them with a ~20├Ś margin (hangar 0.02/22 vs real sky 1.0/1.0); requiring BOTH (plus a coverage gate) keeps it conservative, so a coherent-but-large landscape sky stays flagged. Returns empty when the interior signature matches (`disarm_on_incoherent=False` restores the raw heuristic). Mirrors `AtlasScopeMask`'s self-disarming-fallback pattern.
- **Multi-angle patch projection (`AtlasAddPatchView` + `ProjectionSource` + `camera_math.orbit_camera`):** single-photo projection only textures what the recovered camera saw; occluded/grazing areas go black on orbit. To fill them, an AI novel view of the same scene (generated at a defined angle, e.g. the Qwen-Image-Edit-2511 Multiple-Angles LoRA, `<sks> [azimuth] [elevation] [distance]`) is added as a **patch**. The crux is registration: the LoRA output is an independent generation with no ground-truth transform back to the scene, so the patch camera is **constructed, not solved** ŌĆö `orbit_camera(primary_extrinsics, pivot, d_azimuth, d_elevation, distance_scale)` orbits the recovered camera around the scene pivot (its ground look-at point, `ground_lookat_pivot`) and rebuilds the view matrix via an unambiguous `look_at` (so it shares the primary's Y-up right-handed worldŌåÆcam frame by construction). **The LoRA angles are ABSOLUTE (subject-relative), not relative to the source view** ŌĆö "right side view" is 90┬░ around the *subject's* front, matching the ComfyUI-qwenmultiangle named views (0/45/ŌĆ”/315 azimuth, ŌłÆ30/0/30/60 elevation, close/medium/wide). So `AtlasAddPatchView` takes both the source's named view and the patch's named view and applies the **difference** as the orbit (`patch ŌłÆ source`); a straight-on source is "front view" so the patch's named angle maps directly. GeoCalib can't recover the azimuth delta automatically (single-image yaw is unobservable ŌĆö see `solver._rotation_from_up_vector`), which is exactly why this is a named-view input plus a `flip_azimuth` handedness toggle, calibrated by eye. `AtlasAddPatchView` then derives the patch view's own relief geometry (Depth Anything) *in that constructed frame* and appends a `ProjectionSource` (schema: camera + novel-view `image_b64` + its `proxy_geometry` + priority + angle provenance; JSON-safe, backward-compatible empty default on `LatentScene`). The viewport (`atlas_blockout.js` `buildPatchSources`) builds each source's geometry with its own projection material bound to that source's camera+image, plus a **facing-ratio mask** (`uFacingThreshold`: `|normal┬Ędir-to-camera| < threshold` discards grazing fragments ŌĆö the dot-product occlusion from gs_mptk; primary passes ŌłÆ1 = never discards and keeps priority, patches pass ~0.2 so they only paint head-on surfaces and fall through elsewhere). The facing mask decides *whether* a source can paint a fragment at all; `ProjectionSource.priority` then decides *which* source wins where more than one can ŌĆö `priorityToRenderOrder`/`priorityToOffsetUnits` (same file) convert it into a per-mesh `renderOrder` (deterministic exact-depth ties) plus a small priority-scaled `polygonOffsetUnits` bias (wins only the near-tie band; a genuinely nearer mesh's real depth still wins normal z-test), with the primary given a sentinel `renderOrder` above any patch so it stays implicitly highest without a synthetic priority number. **Load-bearing assumption, verify first:** the LoRA's requested angle must actually correspond to the constructed orbit ŌĆö this is a calibration go/no-go (see `examples/atlas_camera_multiangle_patch_proof_workflow.json`), because both the camera placement and the patch-depth fusion happen in the constructed frame. gs_mptk (the old Maya Mattepainting Toolkit) is the *conceptual* reference for occlusion-masked multi-projection only; the whole pipeline here is Atlas/ComfyUI-native. The named-view dicts and the patch ŌłÆ source delta math (`_AZIMUTH_VIEWS`/`_ELEVATION_VIEWS`/`_DISTANCE_VIEWS`/`_named_view_orbit_delta` in `nodes.py`) are module-level, not `AtlasAddPatchView`-private, specifically so `AtlasOcclusionMask` (below) can place its own target/patch camera identically ŌĆö any drift between the two would silently misalign a precomputed occlusion mask from the patch geometry `AtlasAddPatchView` derives afterward for the same image. `AtlasOcclusionMask` is a pure backend/numpy pre-pass (`depth_geometry.primary_camera_validity_mask`, no browser round-trip ŌĆö deliberately, so it runs headlessly via the ComfyUI API and stays out of `atlas_camera.core`'s "no Three.js dependency" rule) that masks where the PRIMARY camera's projection is invalid at the target/patch view (behind-camera/out-of-frame/too-grazing ŌĆö the same facing-ratio test as `uFacingThreshold` above, just evaluated in Python against the primary instead of in the projection shader against each patch), meant to feed `ImageCompositeMasked` before the result becomes an `AtlasAddPatchView` patch. It is Phase 1 only (frustum/frame/facing-angle); true depth-shadow occlusion (surfaces hidden behind nearer geometry from the primary's view despite passing those tests) is designed but not yet built ŌĆö see `docs/dev/atlas_occlusion_mask_implementation_plan.md`.
- **Camera path animation (`AtlasBlockoutViewport`'s ­¤Äź Camera Path mode + `camera_path.py` + `AtlasExportCameraPathUSD`):** single-photo projection is normally inspected from one static (orbit-clamped) viewpoint; this mode lets an artist test how it holds up under an actual camera *move* ŌĆö orbit/dolly/pan ŌĆö without a DCC. A camera path is a list of `AtlasCameraKeyframe` (`frame_index`, `position`, `target`, `up`, `easing`) plus `fps`/`frame_count` (`schema.AtlasCameraPath`), authored entirely client-side against the same loaded geometry/photo the viewport already has. Authoring uses a **separate, deliberately UNCLAMPED fly controller** (`createFlyControls`, RMB-hold + WASD/QE) rather than the existing orbit control ŌĆö the orbit control's ┬▒80┬░/┬▒55┬░ clamp exists specifically to keep static inspection inside the recovered camera's cone, but leaving that cone is the entire point of testing a move. `camera_path.sample_camera_path` (pure Python, no numpy/torch, mirrors `camera_math.py`'s style) turns keyframes into a full `AtlasExtrinsics` per frame via Catmull-Rom (phantom-duplicated endpoints) + per-segment easing, then `camera_math.look_at_view_matrix` ŌĆö sampling exactly at a keyframe's `frame_index` always reproduces that keyframe exactly, regardless of easing (t=0/1 are fixed points for every easing function used). **What actually moves during preview/bake is the viewport's render/viewing camera ŌĆö never the projection shader's `uAtlasViewMatrix` uniform**, which stays bound to whichever camera (primary or patch) actually photographed the texture; this is the same mental model the existing static orbit-and-inspect already uses (drag to view from elsewhere while the projected texture stays anchored to its real source camera), just driven by a keyframed path instead of a mouse drag. ŌÅ║ Bake Path steps through `0..frame_count-1`, sets the render camera to each sampled pose, calls `renderer.render()`, and reads the canvas directly via `canvas.toDataURL()` (no render-target/flip-Y dance needed ŌĆö unlike the depth/normal/mask passes, this is a plain "what you see is what you get" capture) into `path_frames` (an IMAGE batch feeding a Video Combine node) and `camera_path` (the raw keyframes, feeding `AtlasExportCameraPathUSD` for a time-sampled `.usda`) ŌĆö both written into the same `client_data` widget/channel "Render Passes" already uses, merged rather than overwritten so the two buttons don't clobber each other's results. **Known, deliberate duplication:** the Catmull-Rom+easing math exists in both `camera_path.py` (the source of truth for Bake/USD) and `atlas_blockout.js` (`catmullRom3JS`/`sampleKeyframePoseAtFrame`, for live 60fps Play scrubbing without a Python round-trip) ŌĆö the same kind of accepted duplication as `depth_geometry.py`/`proxy_geometry.py`; keep them in sync by hand. **Inherits, not a new bug:** frames sampled outside the recovered camera's cone render the same documented black/undefined regions as orbiting past it under ­¤ōĮ Project (see "Orbit coverage" below) ŌĆö camera FOV is intentionally NOT keyframed/interpolated in this pass (Atlas intrinsics are fixed per solve, matching every other camera-construction path here); `AtlasCameraKeyframe.fov_deg` is stored for future use but unused by today's sampler/playback. **Presets** (`computePresetEndPose` in `atlas_blockout.js`) give a one-click 2-keyframe quick-start instead of hand-placing keyframes with fly nav every time ŌĆö Pan L/R (rotates the TARGET around the fixed camera position, i.e. the camera swivels in place), Orbit L/R (rotates the POSITION around the fixed target, i.e. the camera arcs around the subject), Dolly In/Out (moves the POSITION toward/away from the fixed target along the view axis). All three derive "left/right" and "in/out" directly from the camera's own forward/right vectors at the moment the preset is applied (`right = cross(forward, world-up)`) rather than from Euler yaw sign, so they're geometrically unambiguous ŌĆö verified numerically (see git history) rather than just reasoned through. Pan/Orbit presets are framed as "arrive at my current vantage from the named side" rather than "leave from here": `PRESET_OPPOSITE` maps each to its mirror (`pan_left`Ōåö`pan_right`, `orbit_left`Ōåö`orbit_right`); applying e.g. Pan Left computes the START pose by running `pan_right`'s transform on the current pose (camera's initial gaze is offset to the right of the geo, since a true pan never moves position) and sets the END to the current pose unchanged, so the move always finishes on whatever you're currently framing. Orbit is the same computation but the offset is positional (start position literally sits to the right of the pivot for Orbit Left) rather than angular, since orbit moves the camera itself. Dolly has no left/right, so it keeps the simpler current-pose-is-the-start shape. Applying any preset replaces the current keyframe list (2 keyframes, `frame_index` 0 and `frame_count-1`, `ease_in_out`) rather than appending, since presets are a fresh quick-start. **FBX camera import** ("­¤ōź Import Camera FBX" in the same panel) reads a DCC-authored camera move (Blender/Maya export) via Three.js's `FBXLoader`/`AnimationMixer`, client-side only (never touching `atlas_camera.core`, same rule `OBJLoader` already follows) ŌĆö parses the uploaded file, finds the first `THREE.Camera` in the FBX with an `AnimationClip`, and samples its world transform at N evenly-spaced times across the clip's duration (`N` = the "samples" input, default 30). An FBX export has no ground-truth relationship to Atlas's world frame ŌĆö the same problem `AtlasAddPatchView` solves for one static offset via a constructed (not solved) patch camera ŌĆö so this treats the whole clip as a **relative** move applied on top of wherever the viewport camera currently is: `alignQuat = Quaternion.setFromUnitVectors(fbxForward_0, baseForward)` aligns the FBX camera's own frame-0 forward to the current base forward (`captureCurrentPose()`, shared with "+ Keyframe" and the presets), then every sample's position-delta-from-frame-0 and forward vector are rotated by that same `alignQuat` before being added onto the base pose ŌĆö verified numerically for pure translation (exact) and pure level-pan (exact angle transfer). A "scale" input (default 1.0) covers the common FBX-cm-vs-solved-meters unit mismatch, adjusted by eye like `flip_azimuth`. **Known limitation:** `alignQuat`'s rotation axis only commutes with the FBX clip's own rotation when both cameras are reasonably level (same assumption `horizon_row_from_extrinsics` already makes) ŌĆö a level FBX pan transfers its exact angle regardless of the two cameras' absolute yaw offset, but a steeply pitched FBX camera aligned to a very differently-pitched base view can pick up rotational "swim" beyond a pure transfer; acceptable for a first pass, recalibrate by eye if it looks off. Imported samples land in `pathFrameCount`/`pathFps` directly (overwriting them to match the import, unlike presets which respect the existing `frame_count` widget) with `easing:"linear"` throughout, then flow through the exact same Play/Bake/`AtlasExportCameraPathUSD` pipeline as hand-authored or preset paths.

- **Nuke camera-projection topology ŌĆö confirmed by actually rendering in Nuke, not by reading documentation (`nuke_exporter.py`, added/fixed 2026-07-07):** `AtlasExportNuke`'s Python-script output (`nuke_projection.py`) had shipped for a while with a node graph that had never actually been executed inside a real Nuke ŌĆö reading its source code looked plausible, but running it (Nuke 16.1v3, Indie license, available on this machine) immediately surfaced real, previously-undiscovered bugs, found by building the graph live via `nuke.createNode()` and checking each node's actual knobs/inputs rather than assuming:

- **Inpaint layers (`AtlasDepthLayerMask` + `AtlasCleanPlateLayer` + `ProjectionSource.metadata["projection_mode"]`):** the classic 2.5D matte-painting move ŌĆö split a solved photo into depth layers, inpaint the regions each layer's foreground occluder hides into a clean plate, then project each clean plate onto its own depth-banded geometry ŌĆö solving the "orbit off the recovered viewpoint ŌåÆ black holes" limitation (see "Orbit coverage" above) for the SAME camera, unlike `AtlasAddPatchView`'s multi-angle patches which fill gaps via novel AI views at OTHER angles and therefore need angle calibration this approach doesn't. **Deliberately reuses `ProjectionSource` rather than inventing new schema:** the viewport's per-source projection material (see "Camera projection" above) already does everything needed ŌĆö these two nodes are orchestration only, following the exact `AtlasAddPatchView` pattern minus the orbit (the camera stays the primary's, unchanged, since a clean-plate layer is a same-camera plate, not a novel angle). `AtlasDepthLayerMask` turns one metric depth band into `(layer_mask, occlusion_mask)` ŌĆö `occlusion_mask` (nearer than the band's near edge) feeds an external `INPAINT_ExpandMask` ŌåÆ `INPAINT_InpaintWithModel` graph to build that band's clean plate; `AtlasCleanPlateLayer` takes the resulting plate image and the same band, clips `build_relief_mesh` to `[near, far]` metres via its new `band_min_m`/`band_max_m` params (the same "exclude the pixel, don't clamp" hole mechanism sky/silhouette exclusion already uses ŌĆö see "Relief mesh tearing" above), and appends a `ProjectionSource` tagged `metadata["projection_mode"]="clean_plate"`.

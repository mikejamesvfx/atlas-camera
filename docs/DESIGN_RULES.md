## Key design rules



- Core schema is pure Python dataclasses — no external deps.

- Optional deps (`numpy`, `cv2`, `fastapi`, `pxr`) are always guarded by try/except with `pip install -e .[extra]` hints in the error message.

- Adapter boundaries must be explicit: coordinate-system conversions are never silent.

- `LatentComponent` slots (`depth`, `geometry`, `lighting`, `semantics`) default to empty until their solvers exist — review packages describe unsupported components rather than omitting them. `depth` is now populated by `solve_still_image_learned(camera_height="auto")` (Depth Anything V2 map summary + measured camera height + confidence); `geometry`/`lighting`/`semantics` remain empty.

- **View-matrix convention (critical):** geometry/camera math must use the full 4×4 `extrinsics.camera_view_matrix` end-to-end (`cam_to_world = inv(view_matrix)`, row-major, column-vector points, translation in column 3) — the convention proven to match the viewport shader. Never build world math from the 3×3 `camera_rotation_matrix` (it has a transpose ambiguity). `core/proxy_geometry.py`, `core/depth_geometry.py`, `core/plane_extraction.py`, `core/room_layout.py`, and `_ground_depth_compute` all follow this; `THREE.Matrix4.set()` consumes the row-major floats verbatim.

- **Multiple geometry-derivation strategies, always artist-selected, never auto-picked:** `AtlasDeriveProjectionGeometry`'s `geometry_mode` picks the output kind — `relief_mesh` (default, a single triangulated mesh following actual depth contours — handles arbitrary/organic shapes but has real gaps at sharp depth discontinuities), `primitives` (simple fitted shapes — cleaner/blockout-like), or `both` (both at once, mostly for comparison; they visually overlap). When `primitives`/`both` is selected, `primitive_method` picks the fitting strategy: `azimuth_walls` (default — vertical walls + foreground boxes/cylinders only, general-purpose but height is clipped to whatever 3D points individually pass a near-vertical-normal filter, so it truncates at a sloped roof/spire/tower — confirmed on real church/tower photos), `ransac_planes` (any-orientation planes via sequential RANSAC seeded by a 2D orientation histogram — best for exterior architecture: roofs, ramps, stepped facades), `room_cuboid` (Manhattan floor+≤4 walls+optional ceiling aligned to the dominant wall direction — best for genuinely orthogonal interiors; produces confidently wrong/skewed results on non-orthogonal rooms, so the artist must pick the right method for the shot rather than relying on auto-detection), `vertical_extrusion` (same wall orientation/distance detection as `azimuth_walls` via the shared `proxy_geometry._cluster_walls_by_azimuth` helper, but height comes from the image-space silhouette instead — the topmost non-sky pixel per column, back-projected at its own depth regardless of local surface normal, per Hoiem/Efros/Hebert's "Automatic Photo Pop-up" (SIGGRAPH 2005) billboard-cutout technique; reaches towers/spires/sloped roofs `azimuth_walls` truncates, at the cost of representing them as a flat vertical plane rather than their true sloped shape). `core/depth_geometry.py` factors the shared back-projection/normals/ground-fit/backdrop logic (bit-for-bit consistent with `proxy_geometry.py`'s own copies, which are left untouched to avoid regression risk) so `proxy_geometry.py` (`azimuth_walls`, `vertical_extrusion`), `plane_extraction.py` (`ransac_planes`), and `room_layout.py` (`room_cuboid`) all agree on world points and metric scale for a given depth map. Every extractor's `stats` dict must include `ground_scale` — required contract with the node's relief-mesh branch, which reuses whichever method's scale was computed. Cross-product normals aren't camera-oriented by construction; `plane_extraction.py`/`room_layout.py` flip them toward the camera globally right after back-projection (sign-invariant under the later ground rescale) so orientation metadata and per-side wall assignment are meaningful.

- **Foreground object height plausibility ceiling (`ProxyDerivationConfig.object_max_height_m`, default 6.0):** `_derive_objects`' foreground-box/cylinder clustering (shared by `azimuth_walls` and `vertical_extrusion` via the same `_derive_objects` call) groups candidate points into objects using a **2D (X,Z) occupancy grid** — it has no notion of height when deciding what belongs to the same object. Found via live browser verification (not unit tests, which all used clean synthetic scenes): on a real photo, a genuine foreground object's footprint spuriously merged with an unrelated tall background structure behind/above it (one that wall-fitting hadn't already claimed) because they shared the same X/Z occupancy cells — the resulting box's height (98th-percentile Y) reflected the tall structure, not the actual foreground object, producing an implausible ~14m "foreground" box that filled the entire viewport edge-to-edge. Mirrors the same "reject an implausible fit" pattern `_fit_cylinder` already uses for `cylinder_radius_range` (returns `None` outside the plausible radius band): when a cluster's fitted height exceeds `object_max_height_m`, refit using only the points at or below that ceiling (dropping the object entirely if too few remain) rather than passing an unbounded height through. Applies to both `derive_projection_proxies` and `derive_vertical_extrusion_proxies` since both call the same shared `_derive_objects`. Does not fix the underlying 2D-only clustering (a genuinely depth/height-aware clustering rewrite is out of scope here, and carries real regression risk for currently-working cases) — this is a plausibility guard on the *output*, not a fix to the clustering algorithm itself.

- **Sky-aware depth (`depth_geometry.detect_sky_mask`):** monocular depth (Depth Anything and similar) has no reliable cue on feature-less sky/cloud regions and often hallucinates noisy, spatially-incoherent depth there rather than a smooth "far" value — left unhandled, that noise gets triangulated into a jagged, spatially-huge mesh chunk (`relief_mesh.build_relief_mesh`) that dwarfs the actual photographed geometry, and can even leak into wall/primitive fitting. `detect_sky_mask(depth, horizon_y=...)` flags a pixel as sky when it's above the solved horizon line AND either at/beyond a far-depth percentile or has high local **roughness** — mean-squared discrete Laplacian, not raw variance — relative to the below-horizon baseline. Roughness (not variance) is deliberate: a genuinely sloped surface (a real roof, a ramp) has a constant local gradient, so its Laplacian is ~0 even though its raw variance in a window is large; plain variance would misclassify real sloped architecture as sky. `build_relief_mesh` uses this to exclude sky from triangulation entirely (a hole, like any other depth discontinuity) rather than merely distance-clamping it; `vertical_extrusion` (above) uses the same mask to find each wall's real silhouette top instead of guessing from noisy sky depth. **Self-disarms on interiors (`_sky_mask_incoherent`, 2026-07-13):** the roughness term is correct for genuine noisy sky but on an INTERIOR also fires on detailed ceilings / far greebled walls above the (arbitrary, sky-free) horizon, punching large scattered holes in real geometry (measured live: a sci-fi hangar lost 39% of its back wall; turning the heuristic off halved the missing fraction). Since the default can't flip globally (it's correct outdoors), `detect_sky_mask` now measures the candidate mask's SHAPE — real outdoor sky is ONE region anchored to the top of frame and vertically contiguous; an interior misfire is scattered fragments with almost nothing top-anchored. Two numpy-only signals (`top_anchored` fraction + runs-per-column) separate them with a ~20× margin (hangar 0.02/22 vs real sky 1.0/1.0); requiring BOTH (plus a coverage gate) keeps it conservative, so a coherent-but-large landscape sky stays flagged. Returns empty when the interior signature matches (`disarm_on_incoherent=False` restores the raw heuristic). Mirrors `AtlasScopeMask`'s self-disarming-fallback pattern.

- **Multi-angle patch projection (`AtlasAddPatchView` + `ProjectionSource` + `camera_math.orbit_camera`):** single-photo projection only textures what the recovered camera saw; occluded/grazing areas go black on orbit. To fill them, an AI novel view of the same scene (generated at a defined angle, e.g. the Qwen-Image-Edit-2511 Multiple-Angles LoRA, `<sks> [azimuth] [elevation] [distance]`) is added as a **patch**. The crux is registration: the LoRA output is an independent generation with no ground-truth transform back to the scene, so the patch camera is **constructed, not solved** — `orbit_camera(primary_extrinsics, pivot, d_azimuth, d_elevation, distance_scale)` orbits the recovered camera around the scene pivot (its ground look-at point, `ground_lookat_pivot`) and rebuilds the view matrix via an unambiguous `look_at` (so it shares the primary's Y-up right-handed world→cam frame by construction). **The LoRA angles are ABSOLUTE (subject-relative), not relative to the source view** — "right side view" is 90° around the *subject's* front, matching the ComfyUI-qwenmultiangle named views (0/45/…/315 azimuth, −30/0/30/60 elevation, close/medium/wide). So `AtlasAddPatchView` takes both the source's named view and the patch's named view and applies the **difference** as the orbit (`patch − source`); a straight-on source is "front view" so the patch's named angle maps directly. GeoCalib can't recover the azimuth delta automatically (single-image yaw is unobservable — see `solver._rotation_from_up_vector`), which is exactly why this is a named-view input plus a `flip_azimuth` handedness toggle, calibrated by eye. `AtlasAddPatchView` then derives the patch view's own relief geometry (Depth Anything) *in that constructed frame* and appends a `ProjectionSource` (schema: camera + novel-view `image_b64` + its `proxy_geometry` + priority + angle provenance; JSON-safe, backward-compatible empty default on `LatentScene`). The viewport (`atlas_blockout.js` `buildPatchSources`) builds each source's geometry with its own projection material bound to that source's camera+image, plus a **facing-ratio mask** (`uFacingThreshold`: `|normal·dir-to-camera| < threshold` discards grazing fragments — the dot-product occlusion from gs_mptk; primary passes −1 = never discards and keeps priority, patches pass ~0.2 so they only paint head-on surfaces and fall through elsewhere). The facing mask decides *whether* a source can paint a fragment at all; `ProjectionSource.priority` then decides *which* source wins where more than one can — `priorityToRenderOrder`/`priorityToOffsetUnits` (same file) convert it into a per-mesh `renderOrder` (deterministic exact-depth ties) plus a small priority-scaled `polygonOffsetUnits` bias (wins only the near-tie band; a genuinely nearer mesh's real depth still wins normal z-test), with the primary given a sentinel `renderOrder` above any patch so it stays implicitly highest without a synthetic priority number. **Load-bearing assumption, verify first:** the LoRA's requested angle must actually correspond to the constructed orbit — this is a calibration go/no-go (see `examples/atlas_camera_multiangle_patch_proof_workflow.json`), because both the camera placement and the patch-depth fusion happen in the constructed frame. gs_mptk (the old Maya Mattepainting Toolkit) is the *conceptual* reference for occlusion-masked multi-projection only; the whole pipeline here is Atlas/ComfyUI-native. The named-view dicts and the patch − source delta math (`_AZIMUTH_VIEWS`/`_ELEVATION_VIEWS`/`_DISTANCE_VIEWS`/`_named_view_orbit_delta` in `nodes.py`) are module-level, not `AtlasAddPatchView`-private, specifically so `AtlasOcclusionMask` (below) can place its own target/patch camera identically — any drift between the two would silently misalign a precomputed occlusion mask from the patch geometry `AtlasAddPatchView` derives afterward for the same image. `AtlasOcclusionMask` is a pure backend/numpy pre-pass (`depth_geometry.primary_camera_validity_mask`, no browser round-trip — deliberately, so it runs headlessly via the ComfyUI API and stays out of `atlas_camera.core`'s "no Three.js dependency" rule) that masks where the PRIMARY camera's projection is invalid at the target/patch view (behind-camera/out-of-frame/too-grazing — the same facing-ratio test as `uFacingThreshold` above, just evaluated in Python against the primary instead of in the projection shader against each patch), meant to feed `ImageCompositeMasked` before the result becomes an `AtlasAddPatchView` patch. It is Phase 1 only (frustum/frame/facing-angle); true depth-shadow occlusion (surfaces hidden behind nearer geometry from the primary's view despite passing those tests) is designed but not yet built — see `docs/dev/atlas_occlusion_mask_implementation_plan.md`.

- **Camera path animation (`AtlasBlockoutViewport`'s 🎥 Camera Path mode + `camera_path.py` + `AtlasExportCameraPathUSD`):** single-photo projection is normally inspected from one static (orbit-clamped) viewpoint; this mode lets an artist test how it holds up under an actual camera *move* — orbit/dolly/pan — without a DCC. A camera path is a list of `AtlasCameraKeyframe` (`frame_index`, `position`, `target`, `up`, `easing`) plus `fps`/`frame_count` (`schema.AtlasCameraPath`), authored entirely client-side against the same loaded geometry/photo the viewport already has. Authoring is **five deterministic one-click move buttons** (2026-07-16 — the manual keyframe editor and the UNCLAMPED free-fly controller `createFlyControls` were removed; git history has them; the orbit controller now stays ENABLED in path mode). `camera_path.sample_camera_path` (pure Python, no numpy/torch, mirrors `camera_math.py`'s style) turns keyframes into a full `AtlasExtrinsics` per frame via Catmull-Rom (phantom-duplicated endpoints) + per-segment easing, then `camera_math.look_at_view_matrix` — sampling exactly at a keyframe's `frame_index` always reproduces that keyframe exactly, regardless of easing (t=0/1 are fixed points for every easing function used). **What actually moves during preview/bake is the viewport's render/viewing camera — never the projection shader's `uAtlasViewMatrix` uniform**, which stays bound to whichever camera (primary or patch) actually photographed the texture; this is the same mental model the existing static orbit-and-inspect already uses (drag to view from elsewhere while the projected texture stays anchored to its real source camera), just driven by a keyframed path instead of a mouse drag. � Bake Path steps through `0..frame_count-1`, sets the render camera to each sampled pose, calls `renderer.render()`, and reads the canvas directly via `canvas.toDataURL()` (no render-target/flip-Y dance needed — unlike the depth/normal/mask passes, this is a plain "what you see is what you get" capture) into `path_frames` (an IMAGE batch feeding a Video Combine node) and `camera_path` (the raw keyframes, feeding `AtlasExportCameraPathUSD` for a time-sampled `.usda`) — both written into the same `client_data` widget/channel "Render Passes" already uses, merged rather than overwritten so the two buttons don't clobber each other's results. **Known, deliberate duplication:** the Catmull-Rom+easing math exists in both `camera_path.py` (the source of truth for Bake/USD) and `atlas_blockout.js` (`catmullRom3JS`/`sampleKeyframePoseAtFrame`, for live 60fps Play scrubbing without a Python round-trip) — the same kind of accepted duplication as `depth_geometry.py`/`proxy_geometry.py`; keep them in sync by hand. **Inherits, not a new bug:** frames sampled outside the recovered camera's cone render the same documented black/undefined regions as orbiting past it under 📽 Project (see "Orbit coverage" below) — camera FOV is intentionally NOT keyframed/interpolated in this pass (Atlas intrinsics are fixed per solve, matching every other camera-construction path here); `AtlasCameraKeyframe.fov_deg` is stored for future use but unused by today's sampler/playback. **The five moves** (⟲/⟳ Orbit L/R, ⇠/⇢ Pan L/R, ⭢ Dolly In — `applyMovePreset` + the kept `computePresetEndPose` vector math in `atlas_blockout.js`) are computed from the **RECOVERED camera pose + the geometry pivot** (`recoveredMoveBasis()`: a throwaway camera posed via `applyRecoveredCamera`, pivot = `lastGeometryPivot` else the ground-point fallback, plus the artist's 🎯 `pivotOffset` — deliberately NOT `targetWithOffset()`, which re-baselines `pivotBase` as a side effect), never from the live orbited camera, so re-clicking a move always produces the identical path for a given solve. Fixed film grammar: **24 fps / 100 frames** (`PATH_FPS`/`PATH_FRAME_COUNT`), 2 keyframes at frames 0/99 with `ease_in_out` on the first. EVERY move's frame 0 sits at the EXACT recovered camera position (the initial 20% positional pre-zoom lerped toward the pivot, which sits above the eye on most plates, and visibly RAISED frame 0 — artist-reported same day, removed): Orbit/Pan look at the pivot (which lies on the recovered central view ray, so composition holds), Orbit arcs 15° about world-Y through the pivot, Pan swivels 15° in place; Dolly In starts at the FULL recovered framing (target ON the recovered view axis at the camera→pivot distance) and pushes in 20% of that distance. The "zoom in a little" is now the **🔭 playback lens** — a slider in the path panel applying a focal MULTIPLIER (default 1.2×, max 2.5×) to the VIEWING camera's FOV during path playback and � Bake ONLY (`playbackLensFovDeg`, read per-frame so mid-preview drags apply live; recorded as `atlas_proxy_path.lens_scale`): never the projection shader's camera, never the solve, never the USD export, all of which keep the recovered intrinsics. The slider's WIDE end is recomputed at every move click (`updateLensWideLimit`) to the focal that frames the projection geometry's bounding sphere about the pivot (`geometryBoundingRadius` — derived + patch groups, EXCLUDING `projection_backdrop`, whose far-plane distance would balloon the wide end; capped at 120° FOV, never above 1.0×), so full-left always shows the geometry's edges. Playback end restores the solved FOV via `applyRecoveredView`; cancelling via the 🎥 toggle restores it explicitly (`solvedFovDeg`). The moves DEPART from frame 0 (the old "arrive at current vantage"/`PRESET_OPPOSITE` framing is gone). A click replaces any existing keyframes, restores 24/100 (undoing an FBX override), persists to client_data, and auto-plays one preview (Play's end-of-playback snap-back restores the recovered view). "Left/right" and "in" still derive from the base pose's own forward/right vectors (`right = cross(forward, world-up)`), numerically verified. **FBX camera import** ("📥 Import Camera FBX" in the same panel) reads a DCC-authored camera move (Blender/Maya export) via Three.js's `FBXLoader`/`AnimationMixer`, client-side only (never touching `atlas_camera.core`, same rule `OBJLoader` already follows) — parses the uploaded file, finds the first `THREE.Camera` in the FBX with an `AnimationClip`, and samples its world transform at N evenly-spaced times across the clip's duration (`N` = the "samples" input, default 30). An FBX export has no ground-truth relationship to Atlas's world frame — the same problem `AtlasAddPatchView` solves for one static offset via a constructed (not solved) patch camera — so this treats the whole clip as a **relative** move applied on top of wherever the viewport camera currently is: `alignQuat = Quaternion.setFromUnitVectors(fbxForward_0, baseForward)` aligns the FBX camera's own frame-0 forward to the current base forward (`captureCurrentPose()` — now used only by this import), then every sample's position-delta-from-frame-0 and forward vector are rotated by that same `alignQuat` before being added onto the base pose — verified numerically for pure translation (exact) and pure level-pan (exact angle transfer). A "scale" input (default 1.0) covers the common FBX-cm-vs-solved-meters unit mismatch, adjusted by eye like `flip_azimuth`. **Known limitation:** `alignQuat`'s rotation axis only commutes with the FBX clip's own rotation when both cameras are reasonably level (same assumption `horizon_row_from_extrinsics` already makes) — a level FBX pan transfers its exact angle regardless of the two cameras' absolute yaw offset, but a steeply pitched FBX camera aligned to a very differently-pitched base view can pick up rotational "swim" beyond a pure transfer; acceptable for a first pass, recalibrate by eye if it looks off. Imported samples land in `pathFrameCount`/`pathFps` directly — FBX import is the ONE sanctioned override of the fixed 24/100 timing (a DCC clip defines its own duration; any move-button click restores the defaults) — with `easing:"linear"` throughout, then flow through the exact same Play/Bake/`AtlasExportCameraPathUSD` pipeline as hand-authored or preset paths.

- **Detached viewport controls (`AtlasViewportControls` + `AtlasBlockoutViewport`'s `controls` input):** by the time Camera Path (presets + FBX import) landed on top of the primitive toolbar, Project/Diagram/Info toggles, and Render Passes, `AtlasBlockoutViewport` had accumulated enough buttons/panels to crowd out the actual 3D view. `AtlasViewportControls` is a companion node with no computation of its own — its single output (`ATLAS_VIEWPORT_LINK`) carries no real data; the graph LINK exists purely so each node's frontend JS can find the OTHER node's live instance (`getLinkedControlsNode`/`getLinkedViewportNodes` in `atlas_blockout.js`, walking `node.graph.links` rather than `app.graph` so this keeps working inside subgraphs). When linked, the viewport's `buildNodeUI` reparents its ALREADY-BUILT `toolbar`/`pathPanel` DOM elements (`mountControls()`) into the Controls node's container instead of its own — every button's `onclick` closure still references the exact same `scene`/`camera`/`pathKeyframes` variables it always did, so this is a pure DOM relocation, not a rewritten API surface, and carries zero behavior-change risk. `mountControls()` re-runs on `onConnectionsChange` (either node) and once more when the Controls node's own container is created, covering both load-order races in a saved workflow. **Backward compatible by construction:** with nothing connected to `controls`, `mountControls()` falls back to the viewport's own container in the exact original DOM order (toolbar, pathPanel, canvas) — so every workflow saved before this split keeps working unchanged. `mountControls()` also short-circuits (`_atlasMountTarget`) if called again with the same target, so repeated `onConnectionsChange` firings never re-append/thrash the DOM. **Resize history — one failed attempt, one working fix, both instructive:** a `node.onResize` handler that called `resizeViewport(w, h)` (changing `canvas.width`/`canvas.height`) to re-render at the dragged size froze the entire tab. Root cause, confirmed by reading the real installed `scripts/domWidget.ts` (extracted from its sourcemap, since it's only documented as "internal, may change"): ComfyUI's own DOM-widget layout math (`BaseDOMWidgetImpl.computeLayoutSize`, upstreamed into litegraph's own node resize handling — `addDOMWidget` already chains its OWN `node.onResize` via `useChainCallback`) sizes the widget from `getMinHeight`/`getMaxHeight`/`getHeight` options (defaulting to a bare 50px minimum if none are given) — nothing here should itself loop. The actual feedback path was CSS: the canvas's `height:auto` meant changing its intrinsic `width`/`height` attributes changed its *laid-out* height too, and neither `container` nor `canvasWrap` had an explicit `height:100%` chain down to the canvas, so dragging the node bigger gave the DOM widget more space that nothing in the chain actually consumed — the widget's real rendered height stayed tied to the canvas's own intrinsic aspect, which is what "resizing resets to a small version" was: not a bug in a resize *handler*, but a layout chain that never grew in the first place. **Fix:** `container` (the addDOMWidget element) now has `height:100%`; `canvasWrap` is `flex:1;min-height:0` (a flex child of `container`, which is `display:flex;flex-direction:column`); the canvas itself is `height:100%` (not `auto`) of `canvasWrap`. `addDOMWidget` also now passes `getMinHeight() { return 240 }` (the sanctioned option) instead of leaving LiteGraph's fallback-50px floor unset. Critically, **no JS resize hook is involved at all** — the canvas's own `width`/`height` attributes (intrinsic WebGL buffer / render resolution, still governed solely by the `resolution` widget) are never touched by resizing; the browser just rescales the existing buffer via ordinary CSS layout, which cannot feed back into anything. `resolution` remains the only way to change actual render resolution; dragging the node just changes displayed size (may blur if dragged far past the current resolution). **Follow-on bug from that same fix:** giving `canvasWrap`/`container` real flex sizing (`flex:1`, `height:100%`) surfaced a second, well-known CSS gotcha — flex items default to `min-width:auto`, which resolves to their content's min-content size, and for a `<canvas>` (a "replaced element" like `<img>`/`<video>`) that min-content floor is its INTRINSIC width (the `width` *attribute*, e.g. 768px — `width:100%` in CSS only affects the *used* size, never this floor). That silently forced `canvasWrap`, and the node containing it, to never shrink below the canvas's intrinsic pixel width regardless of the node's actual dragged size, surfacing as the node snapping/stretching wider on the first interaction that triggered a relayout (mousedown to orbit). Fixed with the standard remedy: explicit `min-width:0` on both `canvasWrap` and `container`, overriding the `auto` default. **Third follow-on, found via ShotCam testing:** `width:100%;height:100%` on the canvas has no aspect-ratio awareness — it always stretches the WebGL buffer to exactly fill its box, which was invisible before because the box's shape and the render resolution's aspect were always coincidentally close (no prior workflow set a render resolution far from the node's on-screen shape). `AtlasDefineShotCam`'s `shot_cam` (see "ShotCam" below) can conform the render resolution to a very different aspect (e.g. 2.39:1 anamorphic) than whatever shape the node happens to be on screen, making the stretch/squash obvious. Fixed with `object-fit:contain` on the canvas (plus a matching `background:#111` on `canvasWrap` so the resulting letterbox bars read as intentional, not a gap) — pure CSS, no new JS resize logic, verified live via `getBoundingClientRect()`/`getComputedStyle` before/after. **Known limitation, not fixed in this pass:** the diagram (📊)/HUD (ℹ) SVG overlays are still sized to the full `canvasWrap` box (`inset:0`,`width/height:100%`), so they'll misalign with the now-letterboxed canvas content whenever a significant aspect mismatch triggers letterboxing — narrow (only visible with those toggles on AND a shot cam far from the node's on-screen aspect), left for a follow-up rather than risking a flexbox+`aspect-ratio` rewrite in a spot with three prior documented bugs.

- **`scene_type` convenience preset, now the primary control (still artist-selected, not auto-detected):** `AtlasDeriveProjectionGeometry`'s `scene_type` widget (default `manual`) grew from 4 to 9 options — `organic`, `mountains`, `forests`, `aerial`, `indoor`, `outdoor`, `simple_walls`, `towers_spires` — specifically to close the "primitive_method is silently ignored when geometry_mode=relief_mesh" confusion: each option is a complete, self-consistent bundle of `geometry_mode`/`primitive_method`/`relief_quality`/`depth_edge_rel`/`max_objects`/`depth_model`, resolved in a class-level `_SCENE_TYPE_PRESETS` dict at the top of `derive()` — purely a convenience mapping onto the exact same code paths, never a new solving strategy. `mountains`/`forests` are both `relief_mesh` at `relief_quality="high"` (terrain/canopy need more grid density than the default to read as continuous); `forests` additionally relaxes `depth_edge_rel` to `1.0` since dense canopy depth is genuinely noisy at small scale and the default threshold shreds it into holes (see the relief-mesh-tearing entry above — this is that same finding, applied). `aerial` is `geometry_mode="both"` + `azimuth_walls` + `max_objects=6` — buildings as boxes over a relief-mesh ground/treeline, the drone/top-down case. `simple_walls`/`towers_spires` newly expose `azimuth_walls`/`vertical_extrusion` as named presets (previously only reachable by manually setting `geometry_mode=primitives` — `scene_type` couldn't reach them at all before). `manual` (default) leaves every widget exactly as set — fully backward-compatible; all-new option VALUES were *added*, none renamed, so workflows saved before this expansion keep working unchanged.

  **Widget visibility follows the preset** (`atlas_camera/comfy/web/atlas_derive_geometry.js`, extension `AtlasCamera.DeriveGeometryPresets`, independent of `atlas_blockout.js`): picking anything but `manual` hides whichever widgets that preset has made moot, via `computeHiddenWidgets()` — always `geometry_mode`/`primitive_method` for any non-manual preset, plus that preset's own explicit override keys, plus `relief_grid`/`relief_quality`/`depth_edge_rel` whenever the resolved `geometry_mode` is `primitives` (no relief mesh ever builds) or `relief_quality` is explicitly set (it always supersedes `relief_grid`), plus `max_objects`/`max_walls` whenever `geometry_mode` is `relief_mesh`-only (objects/walls still get derived internally for `ground_scale`/backdrop, per the design rule above, but are discarded from the kept geometry). A widget only disappears when changing it would have *zero effect* — everything left visible still does something. `SCENE_TYPE_PRESETS` is hand-mirrored in that JS file (same accepted-duplication pattern as `camera_path.py`/`catmullRom3JS` — keep both in sync by hand) purely to decide what to hide; Python's `_SCENE_TYPE_PRESETS` remains the only place the actual override logic runs. Hiding uses the standard litegraph trick — swap `widget.type` to an unrecognized string and `computeSize` to `[0,-4]`, restoring both on re-show — never touches `.value`, so serialization/queueing behave exactly as if the widget were visible.

  The preset only reaches this node's own depth estimation; if `AtlasLearnedSolveFromImage` upstream uses `height_mode=measure_from_depth`, its separate `depth_model` widget must be set to match by hand (nodes don't cross-configure each other's widgets). Recorded in `debug_metadata["proxy_derivation"]` (`scene_type`, `depth_edge_rel`, `relief_grid`, `relief_quality`, `max_objects`) for traceability.

- **Composable geometry derivation (`AtlasDepthMap` + `AtlasDeriveReliefMesh`/`Walls`/`TowersSpires`/`RoofsFacades`/`InteriorRoom` + `AtlasMergeGeometry`):** the Nuke-style alternative to `AtlasDeriveProjectionGeometry`'s `scene_type` presets, for scenes that genuinely mix strategies (foreground buildings over background terrain, say) rather than fitting one named preset. Instead of one node with a combo that locks/hides other widgets, each derivation strategy is its own node with *only* parameters that always apply — no `geometry_mode`/`primitive_method` cross-product exists to be confusing, so none of these nodes need `atlas_derive_geometry.js`'s widget-hiding trick. `AtlasDeriveProjectionGeometry` itself is untouched — this is additive, not a replacement, and the two private helpers these new nodes share (`_solve_camera_params`, `_horizon_y_from_solve`, `_depth_map_for_solve`, `_replace_proxy_role_geometry` in `nodes.py`) intentionally are **not** used to refactor the existing node, to avoid risking regression in tested, shipped behavior.

  **Depth is shared, not just for speed.** `AtlasDepthMap` runs Depth-Anything once and passes the raw `DepthResult` by reference as a new `ATLAS_DEPTH_MAP` type (same zero-serialization custom-type pattern as `ATLAS_SOLVE`/`ATLAS_CAMERA_PATH`/`ATLAS_VIEWPORT_LINK` — an arbitrary string type name, Python object passed in-process, no JSON round-trip). Every extraction strategy (`build_relief_mesh`, `derive_projection_proxies`, `derive_vertical_extrusion_proxies`, `extract_planes_ransac`, `extract_room_cuboid`) shares one contract — 2D float64 `(H,W)` forward-Z array + `view_matrix`/`fx,fy,cx,cy` — confirmed identical across all five call sites, so one `AtlasDepthMap` output safely fans out to several derive nodes. This is required for **correctness**, not just efficiency: each strategy independently fits its own ground plane from whatever depth map it receives, so two branches estimating depth separately could disagree slightly on metric scale, landing their ground planes at different heights and misaligning the merged result. Distinct from `AtlasDepthAnything` (existing): that node's IMAGE output is a lossy, per-image min-max-normalized preview — the real near/far distances and `is_metric` flag are computed then irrecoverably discarded — so it cannot be repurposed as the metric carrier.

  **Chaining derive nodes silently loses data — this is why the merge node exists.** Every derive node (new and the original `AtlasDeriveProjectionGeometry`) deep-copies its input solve and then strips *all* prior `PROXY_ROLE`-tagged geometry before adding its own (`_replace_proxy_role_geometry`) — deliberate, so re-running a derive node never piles up stale geometry across repeated executions. The side effect: wiring `AtlasDeriveWalls`'s output solve directly into `AtlasDeriveReliefMesh`'s `solve` input does **not** accumulate both branches' geometry — the second node erases the first's. `AtlasMergeGeometry` is the one explicit, visible place two branches' `proxy_geometry` lists actually combine — binary, matching Nuke's own Merge node, chainable for 3+-way combination rather than a fixed N-ary input cap. It takes `solve_a`'s full deep copy as the base and appends **only `solve_b`'s `PROXY_ROLE`-tagged geometry** (what `solve_b`'s own derive node actually added), never `solve_b`'s full `proxy_geometry` list. That distinction was found via live end-to-end verification, not reasoned out upfront: every solve used to inherit a placeholder `"ground_plane"` entry (`role="ground"`, a *different* role, not `PROXY_ROLE` — see the "two other unrelated `role`-key conventions" note that governs `proxy_geometry.py`'s own filtering) from `projection_scene.create_default_projection_scene()`, and every derive node passed that pre-existing entry through untouched — so naively appending `solve_b`'s entire list duplicated it on top of `solve_a`'s own copy of the exact same thing, even though `solve_a` already provided it. Filtering `solve_b`'s contribution to `role == PROXY_ROLE` fixes this in one place for any current or future non-`PROXY_ROLE` pass-through geometry, not just this one case — kept even after the specific placeholder below was removed, since it's the correct general contract regardless. It also **dedupes `"projection_backdrop"`** within that same `PROXY_ROLE` set: every extraction strategy always emits exactly one primitive with that literal name (a far catch-all plane), and `atlas_blockout.js` has a name-based special case for it — merging two branches that each derived one would still produce two overlapping backdrop planes without this, so the merge keeps only the first one encountered (`solve_a`'s). No viewport-side changes were needed otherwise: `buildDerivedProxies` already treats `proxy_geometry` as one flat, source-agnostic list. Verified live against a real photo (`AtlasLearnedSolveFromImage` → `AtlasDepthMap` → `AtlasDeriveWalls` + `AtlasDeriveReliefMesh` → `AtlasMergeGeometry` → `AtlasExportSolveJSON`, comparing the merged JSON against each branch exported alone) and then smoke-tested across 5 more varied photos: merged geometry always carried exactly one `ground_plane`/one `projection_backdrop` (never duplicated), each branch's own contribution intact (walls/boxes/cylinders varying sensibly per photo, relief mesh always present), no NaN/Infinity in any output.

- **The "ground" naming confusion, resolved by removing the actual dead weight (`projection_scene.py`):** the live verification above surfaced a second, more interesting issue than a naming collision — `create_default_projection_scene()` (called by several `solver.py` solve-construction paths) used to seed every solve with a placeholder `"ground_plane"` primitive (`role="ground"`, a 10×10m flat default). Exhaustive grep across the whole repo (Python, the React `ui/`, JS) found **zero consumers**: `serialize_proxy_geometry` only ever sends `role=="projection_proxy"` (`PROXY_ROLE`) primitives to the viewport, so it was never rendered; no exporter reads it; `up_axis`/`coordinate_system` are already their own fields on `AtlasProjectionScene`, so even its metadata was redundant. `proxy_box`/`add_axis_guides` in the same module are equally unused outside their own test file (left alone — out of scope for this specific fix, since they're unused-but-harmless rather than actively confusing). Its one real, observed effect was confusion: an almost-identically-named, *actually rendered* `"projection_ground"` primitive (`PROXY_ROLE`, from `derive_projection_proxies`/`AtlasDeriveWalls`) sits in the same list, and the placeholder was the root cause of the `AtlasMergeGeometry` duplication bug above. Fixed by deleting the placeholder from `create_default_projection_scene()` entirely — rather than renaming it — since nothing needs it; a solve's `proxy_geometry` now starts genuinely empty. If a real default-ground visualization is ever wanted again, it should be built as a proper `PROXY_ROLE` primitive by whichever node actually wants to show it, not injected invisibly at solve-construction time.

  **Known, documented risk, not fixed in this pass:** `derive_projection_proxies` (azimuth_walls) carries its own inline copy of the ground-fit/scale math rather than calling the shared `depth_geometry.fit_ground_and_scale` that `derive_vertical_extrusion_proxies`/`extract_planes_ransac`/`extract_room_cuboid` all call — confirmed line-for-line identical arithmetic today (bit-identical results on identical inputs), but it's two independently-maintained copies that happen to agree, not one shared function. Sharing one `ATLAS_DEPTH_MAP` across branches removes the "different depth estimate per branch" risk entirely; if these two copies ever drift apart in a future edit, cross-strategy merges could silently misalign. Pre-existing, out of scope for this feature.

- **ShotCam — a project-level render/output camera format (`AtlasShotCam` + `AtlasDefineShotCam` + `AtlasMergeGeometry`'s/`AtlasBlockoutViewport`'s `shot_cam` input):** every `AtlasSolve` carries its own `AtlasIntrinsics` derived from whatever photo was actually solved — its own resolution, aspect ratio, guessed sensor/lens. There was no concept above a single per-image solve, which becomes a real gap once a graph combines geometry from more than one source image (`AtlasMergeGeometry`'s own docstring already admitted this: *"does not check for or correct a mismatch between solve_a and solve_b"*), and `AtlasBlockoutViewport` always derived its render resolution/aspect from whatever `source_image` happened to be wired in. `AtlasShotCam` (schema.py) is a small, JSON-safe, intrinsics-only dataclass — `sensor_width_mm`/`sensor_height_mm`/`focal_length_mm`/`resolution_long_edge_px` — analogous to a Nuke/Resolve project format setting: it describes the FINAL render/export camera, decoupled from any individual solved photo's own sensor/lens/aspect. Deliberately has no position — camera placement always still comes from whichever solve's own recovered extrinsics is already in play. `atlas_camera.core.intrinsics.intrinsics_from_shot_cam()` turns one into real pinhole `AtlasIntrinsics` via the existing `build_intrinsics()`/`focal_length_mm_to_pixels()` (no new pinhole math). `AtlasDefineShotCam` is a pure config-carrier node (same spirit as `AtlasViewportControls`), output type `ATLAS_SHOT_CAM` (same zero-serialization in-process pattern as `ATLAS_SOLVE`/`ATLAS_DEPTH_MAP`).

  **Two entry points, one precedence rule.** `AtlasMergeGeometry`'s optional `shot_cam` input, when connected, is a pure *attachment* — `out.shot_cam = shot_cam` — never a mutation of `out.camera` (which stays `solve_a`'s own intrinsics/extrinsics exactly as before). This lets a project format defined once flow downstream to whatever consumes it, without rewiring. `AtlasBlockoutViewport` also takes an optional direct `shot_cam` input; its precedence is resolved once at the top of `render()`: an explicitly-wired `shot_cam` wins, else `solve.shot_cam` (inherited from an earlier merge) is used if present, else today's `_fit_long_edge(source_image aspect)` behavior — fully backward compatible when neither is present. Verified live end-to-end via both paths against real solved data: a direct wire and a merge-inherited attachment both correctly conformed independent `AtlasBlockoutViewport` instances to the identical shot format.

  **Why this is safe — and the one place the naive implementation would have broken things:** geometry is world-space and metric; it doesn't care about sensor/lens format, so ShotCam never needs to touch any derive node. The one real hazard, found and fixed during implementation (not anticipated in the original design): `_extract_blockout_camera`'s top-level `fx`/`fy`/`cx`/`cy`/`image_width`/`image_height` are read by TWO different frontend consumers from the exact same payload object — `applyRecoveredCamera` (the live orbit/viewing camera's FOV/aspect) AND `makeProjectionMaterial` for the PRIMARY source's own texture-sampling uniforms (`applyCamera(data)` and `setProxies(data)` in `atlas_blockout.js` are both called with the identical `data` object from the same execution). Naively overwriting `fx`/`fy`/`cx`/`cy` to conform to a shot cam would have silently corrupted how the actual photo gets projected onto geometry. The fix: two new, genuinely separate payload keys, `render_fy`/`render_image_height`, read ONLY by `applyRecoveredCamera` (`atlas_blockout.js`, one small edit — `data.render_fy ?? data.fy`, `data.render_image_height ?? data.image_height`, defaulting to today's values when no shot_cam is present) — `fx`/`fy`/`cx`/`cy`/`image_width`/`image_height` themselves are never touched, so `makeProjectionMaterial` for the primary (and every patch's own `ProjectionSource` camera, entirely unrelated to any of this) keeps sampling correctly regardless of what shot format is active. `target_width`/`target_height`, by contrast, needed no separate key at all — they're already routinely decoupled from `image_width`/`image_height` today (`_fit_long_edge` resizes the canvas to `resolution` regardless of the source photo's actual pixel dimensions), and `makeProjectionMaterial` never references them, confirmed by reading its exact uniform list. `camera_meta` (the ℹ Info/📊 Diagram diagnostics — `focal_mm`, `fov_h_deg`, etc.) is also left untouched by a shot cam, since those describe how the photo was actually shot, not the output format.

  **Deferred, not in this pass:** none of `AtlasExportUSD`/`AtlasExportReliefMesh`/`AtlasExportMayaReviewScene` currently has any resolution/format override — camera params come straight from `solve.camera`/`intr.image_width`/`intr.image_height` in all three. Extending the same `shot_cam`-input pattern to these exporters is a structurally identical follow-on, deliberately out of scope here to keep the first pass small and reviewable.

- **Nuke camera-projection topology — confirmed by actually rendering in Nuke, not by reading documentation (`nuke_exporter.py`, added/fixed 2026-07-07):** `AtlasExportNuke`'s Python-script output (`nuke_projection.py`) had shipped for a while with a node graph that had never actually been executed inside a real Nuke — reading its source code looked plausible, but running it (Nuke 16.1v3, Indie license, available on this machine) immediately surfaced real, previously-undiscovered bugs, found by building the graph live via `nuke.createNode()` and checking each node's actual knobs/inputs rather than assuming:

  1. `Card3D["xsize"]`/`["ysize"]` **do not exist** — `Card3D` is a lens/format-driven camera billboard (sized via its `format`+`lens_in_focal`/`lens_in_haperture` knobs to auto-fill a camera's frustum at a given depth), not a manually-sized flat plane. The plain **`Card`** node is the right one for an arbitrary-sized ground plane: it's a 1×1 unit quad by default (`p0`=[-0.5,-0.5]..`p3`=[0.5,-0.5], confirmed by inspecting its knobs live), sized via the universal `scaling`/`uniform_scale` transform knobs every 3D node shares — `card["scaling"].setValue([40.0, 40.0, 1.0])` for a 40×40m plane, same `rotate=[-90,0,0]` to lie flat as before.

  2. `ScanlineRender["format"]` **does not exist** — render/output resolution is a `Root` (project) setting; the fix sets `nuke.root()["format"]` instead of a per-node knob.

  3. **The whole topology was backwards.** `Project3D2` is genuinely a **2-input node** (`maxInputs()==2` fresh out of `createNode`) — img and cam only, with *no geometry input at all*; "projecting" means baking the camera's view into the 2D image stream, not stamping texels onto incoming geometry. The projected 2D output then feeds *into* the **Card's own image input** (`card.setInput(0, proj)`) — the Card is what actually carries that image into 3D space, not the reverse. And `ScanlineRender`'s three inputs are **not** obj=0/cam=1 as its own "3 inputs" might suggest from a naive reading — `canSetInput()` probing confirmed the real mapping is **bg=0 (unused here), obj=1, cam=2**; the old code's `render.setInput(0, proj); render.setInput(1, cam)` silently connected nothing at all (`canSetInput` returns `False` for both attempts at those indices). Corrected: `render.setInput(1, card); render.setInput(2, cam)`.

  4. **Windows paths silently corrupt** when passed through `Node["file"].setValue(a_path)` — Nuke's live knob-setting API runs the value through TCL escape interpretation regardless of how the Python string was built, and backslash-letter sequences (`\Users`, `\AtlasCamera`, ...) get eaten outright (confirmed: a raw `r"C:\Users\..."` path arrived at the Read node as `"C:Users<name>Desktop..."` — same corruption independently confirmed on a `Write` node's own `file` knob). Forward slashes are accepted identically on Windows and dodge the whole TCL-escaping question entirely, so both the primary and package-fallback source paths are now normalised (`.replace("\\", "/")`) before being set.



  Each fix was verified by actually rendering the corrected graph end-to-end (`nuke.execute()` through a real `Write` node) and diffing output file sizes across iterations to confirm pixel-identical results — not just "no exception raised". The corrected topology now also backs a genuinely new deliverable: `write_nuke_native_script()` writes a plain-text **`.nk`** scene (Root/StickyNote/Read/Camera2/Project3D2/Card/ScanlineRender, no Python execution needed — File > Open or drag-and-drop ready) describing the identical graph, wired as `AtlasExportNuke`'s new second output (`nk_path`, additive — `script_path` stays at output slot 0, so the 3 existing example workflows already using this node are unaffected).

  **The one true text-serialization limitation found, and its fix:** `.nk`'s `push`/`pop` stack model resolves a pushed connection via LIFO order (last `push` considered first) and, for each candidate, walks a node's inputs from 0 upward for the first empty type-compatible slot — reverse-engineered empirically by round-tripping small graphs through `nuke.scriptReadFile()` (chosen over `nuke.scriptOpen()`/`nodeCopy()`/`scriptSave()`, all three of which are encrypted under this machine's Nuke *Indie* license tier and unusable as a plain-text oracle) and inspecting the resulting connections. That model reliably wires `Project3D2` (img/cam) and auto-wires the geometry node's single input, but **reusing the same `Camera2` a second time** as a push target — already consumed once by `Project3D2` — never reliably re-resolves into `ScanlineRender`'s cam slot via text alone, confirmed across every push-order and pairing permutation tried. Rather than fight that one case further, `ScanlineRender`'s Camera2 link is completed by a one-line Python callback on `Root`'s `onScriptLoad` knob (`nuke.toNode('ScanlineRender1').setInput(2, nuke.toNode('Camera1'))`), which Nuke runs automatically the instant the script opens — still one self-contained `.nk` file, no companion script, and confirmed (via `scriptReadFile`, which turns out to honor `onScriptLoad` itself) to produce byte-identical render output to the `.py` path and to a hand-built live-API graph.

  **Real relief mesh instead of the flat ground card (`relief_mesh_obj_path`, added 2026-07-07):** both writers take an optional `relief_mesh_obj_path` — wire `AtlasExportReliefMesh`'s `obj_path` into it (same established pattern as `AtlasExportMayaReviewScene`'s own `relief_mesh_obj_path` input) to import the actual derived geometry via **`ReadGeo2`** (confirmed live: natively reads both `.obj` and `.fbx`, though `atlas_camera` only writes OBJ today — no FBX writer exists in the codebase) instead of the default flat `Card`. Both a `Card` and a `ReadGeo2` are single-image-input nodes (`maxInputs()==1`), so the swap is a drop-in replacement in the existing `Project3D2 → geo.setInput(0, proj)` topology — the shared variable is named `geo`/`Geo1` in the generator code specifically so neither writer needs to know which one it's dealing with. **Deliberately still live-projected, not a static UV-baked texture, even though the relief mesh's own vertex UVs already bake that exact camera projection** (confirmed live: feeding the raw, un-projected source photo directly into `ReadGeo2`'s image input renders pixel-plausibly from the ORIGINAL recovered camera's own viewpoint, since the UVs alone already encode which pixel maps to which point) — Project3D2 is kept in the chain anyway because a static bake doesn't behave like a real projection rig: it can't correctly black out geometry outside the recovered frustum (the same matte-painting property the ComfyUI viewport's own 📽 Project relies on), and swapping in a different plate wouldn't re-project. **A second, independent gotcha found via `ReadGeo2` specifically** (distinct from the Card case): Nuke's 3D pipeline does **not** auto-apply an imported OBJ/MTL's own `map_Kd` texture reference at all — a `ReadGeo2` with nothing wired into its own image input renders as pure black ("no valid channels" on the eventual `Write`), regardless of a perfectly valid MTL sitting next to the OBJ; the texture must always be fed in explicitly via a real input connection, which the live-projection wiring above already provides for free. **Path portability:** `AtlasExportReliefMesh`'s `obj_path` output is relative to ComfyUI's own working directory (same convention as every export node's `output_dir`), not wherever an artist eventually launches Nuke from — `AtlasExportNuke.export()` resolves it to an absolute path before handing it to either writer. Verified end-to-end via the live ComfyUI queue (`AtlasRegisterPlate`/`AtlasAttachSourcePlate` → `AtlasExportReliefMesh` → `AtlasExportNuke.relief_mesh_obj_path`) and by loading the exact generated `.nk` in real Nuke: renders the actual hangar geometry (real silhouette tearing at depth discontinuities, not a flat floor), pixel-plausible match to the flat-Card render from the same recovered viewpoint.

- **Relief mesh tearing on noisy AI-image depth (`relief_grid`/`depth_edge_rel`/`relief_quality`):** `build_relief_mesh` tears a quad into a hole whenever its corner depths differ by more than `depth_edge_rel` (relative) — this is *load-bearing* (silhouettes must not rubber-sheet onto the background), but a genuinely noisy monocular depth estimate (common on complex/reflective/low-light real photos) can trip that threshold across large swaths of the mesh even where there's no real discontinuity, producing a visually "shattered"/shard-like result rather than a continuous surface — confirmed empirically (not just reasoned through): on a synthetic depth map with exactly one real edge, `torn_fraction` drops monotonically as `grid_long_edge` rises (32→256 took it from 0.040 to 0.006), because a finer grid means each quad spans less real-world area and is less likely to straddle spurious per-pixel noise. `AtlasDeriveProjectionGeometry`'s `relief_grid` (default **128**, range 16–1024 — was 96/256 before this was diagnosed; now matches `AtlasExportReliefMesh`'s own default) and new `depth_edge_rel` widget (default 0.5, same param/default as `AtlasExportReliefMesh` — previously hardcoded, not exposed on this node at all) are both wired straight into that same `build_relief_mesh` call. `relief_quality` (`custom`/`low`/`medium`/`high`/`ultra`, default `custom`) is a `relief_grid` quick-pick — `low=64, medium=256, high=512, ultra=1024` — following the exact same convenience-preset pattern as `scene_type` above (a class-level `_RELIEF_QUALITY_PRESETS` dict resolved at the top of `derive()`, `custom` leaves `relief_grid` exactly as its own widget says, fully backward compatible). `ultra` trades a much heavier mesh (bigger solve JSON, slower viewport) for the fewest holes — recommend trying `high` first on troublesome photos. Both `relief_grid` and `relief_quality` are recorded in `debug_metadata["proxy_derivation"]` for traceability.

- Output dimensions **auto-adopt the source image** — never hardcoded. The solve carries `camera.intrinsics.image_width/height`; `AtlasGroundDepthMap`/`GroundMask`/`HorizonMask` take `image_width`/`image_height` where **0 = auto** (via `_solve_image_size`), and `AtlasBlockoutViewport` takes a single `resolution` (long edge) and derives W×H from the source aspect (via `_fit_long_edge`). The blockout frontend resizes its canvas/camera to those `target_width/target_height` on execution, so the viewport inherits the image aspect.

- Camera height / metric scale is **measured, not assumed**, via a tiered cascade (best evidence first), each adopted only above its confidence threshold and otherwise surfaced as a flagged candidate (never silently promoted — matches the LLM-suggestion confirm principle):

  1. **Reference object** (`scale_references` / `AtlasReferenceScaleSolve`) — known-size object (person/door/car from `reference_data`, or explicit height) solved by single-view geometry. Most reliable; `_REFERENCE_ADOPT_CONFIDENCE`. VLM cues auto-feed this tier: `AtlasVLMScaleCues` → `scale_references` JSON → `AtlasApplyScaleReferences` (which rescales **only on `confirm`** — `multimodal_helper.scale_references_from_observation()` maps `SceneScaleCue`s to specs via the registry; LLM cues are never auto-promoted).

  2. **Depth ground-plane** (`camera_height="auto"`) — Depth Anything V2 + `estimate_ground_height_from_depth`; `_HEIGHT_ADOPT_CONFIDENCE`. Unreliable on AI imagery, so usually low-confidence.

  3. **Assumed default** — last resort, flagged. `solve.debug_metadata["scale_source"]` records which tier won.

- **`hole_mask` — the mesh's own gap data, surfaced (2026-07-08):** `ReliefMesh.hole_mask` ((H,W) bool, ALWAYS full source-image resolution regardless of grid decimation) captures exactly where 📽 Project will show black: the per-pixel exclusion seed (sky/invalid/band) OR'd with a vectorized rasterization of every torn quad (`np.searchsorted` pixel→quad gather — no Python loop even at grid 1024). Exposed as a MASK output on `AtlasDeriveReliefMesh`, `AtlasCleanPlateLayer`, `AtlasDeriveProjectionGeometry` (zero mask in primitives mode), and opt-in on `AtlasDepthLayerMask` (`compute_hole_mask` — a real duplicate mesh build, deliberately, so the signal exists BEFORE the external inpaint runs and can drive it; `AtlasCleanPlateLayer`'s own copy is necessarily post-hoc QA). The naive `occlusion_mask` ("is something nearer") is blind to tears INSIDE a band; the two are separate outputs, unioned explicitly by the artist — never auto-combined. The mask node's hole build deliberately does NOT apply the layer's edge overhang: a pessimistic hole_mask over-inpaints safely, an optimistic one under-inpaints.

- **An explicit `exclude_mask` REPLACES the internal sky heuristic (2026-07-08, found live):** `detect_sky_mask` flags above-horizon far/rough pixels — which eats real tall geometry (37% of monument valley's butte silhouettes were heuristic-excluded despite a perfect SAM sky mask wired in; holes inside the buttes collapsed 0.52→0.096 with the fix). All four relief-mesh node call sites pass `apply_sky_heuristic = exclude_mask is None`; `_metric_depth_and_validity` does the same for band masks. Need both signals? OR the sky into the exclusion mask externally. `exclude_mask` inputs exist on `AtlasDeriveReliefMesh`/`AtlasDeriveProjectionGeometry`/`AtlasDepthLayerMask`/`AtlasCleanPlateLayer` (any resolution, nearest-resized).

- **Per-pixel edge mattes + boundary overhang — the complete DMP edge doctrine (2026-07-08):** geometry silhouettes tear at grid-quad resolution (~30px steps at grid 128 on 4K) and always will; the fix is never more tessellation. (1) `ProjectionSource.mask_b64` (lossless PNG data URI — a matte's 0.5 threshold must not eat JPEG ringing) is sampled by the projection shader at the SAME projected uv as the photo (`uMatte`/`uHasMatte`; loaded with `NoColorSpace`, NOT sRGB — a gray 128 would decode to ~0.216 and shift the threshold) and discards outside the matte. (2) A matte can only CUT geometry that exists: `build_relief_mesh(edge_overhang_cells=N)` extends valid grid cells outward copying neighbor depth (flat skirt) so boundary quads survive for the matte to cut — verified live: 40px skyline-strip uncovered fraction 0.475→0.001. NEVER overhang unmatted (visible stretched texels): `AtlasCleanPlateLayer` passes `2 if embed_matte else 0`. `AtlasSkyDomeLayer` embeds its SAM mask automatically; `AtlasCleanPlateLayer.embed_matte` auto-computes the band matte (in-band ∪ filled footprint when `fill_occluded`, minus exclude) with a `layer_matte` MASK override. Mattes ride into both DCC exports (plate ALPHA + standalone `{layer}_matte.png`).

- **Deterministic sky extension — edge-extend + frame outpaint, not inpainting (2026-07-08):** for narrow disocclusion slivers of smooth gradient sky, the classic Nuke premult→dilate edge-extend is indistinguishable from an inpaint at a fraction of the cost. `AtlasSkyDomeLayer.edge_extend_px` (default 48) smears sky edge colors past silhouettes (`_extend_edge_colors`, pure numpy at quarter res — sky is low-frequency), dilates the embedded matte to expose the extension on disocclusion, and scales the dome's mesh overhang. `frame_outpaint_px` (default 64) applies the same trick past the FRAME edges: pad the plate canvas edge-replicated, and give the sky source its OWN widened camera (cx/cy+P, W/H+2P, same fx/fy/pose — per-source cameras make this free; primary untouched; the Nuke export's `layer_focal_mm` self-corrects to the wider FOV). LaMa/inpaint chains remain the right tool for large STRUCTURED reveals (clouds behind buildings) via `plate_image`.

- **Ground-anchored extrusion (`ground_anchor`) + roofline segmentation (`roofline_split`) — user-proposed, 2026-07-10:** monocular depth "bananas" tall structures, but the ray DIRECTION through a pixel is exact — so where a building visibly meets the analytic Y=0 ground plane, its footprint is pure geometry (ray∩ground), the depth model is demoted to grouping pixels, and anchored buildings SIT on the ground (extrude from Y=0) with **heights from ray∩anchored-plane** (the silhouette top pixel is reliable; its own back-projected depth is not — warped far depth inflated real rooftops to 140m). Everything hard about this feature is in its FOUR gates, each found by a real failure: (1) **base pool ≠ fit slab** — a banana-biased plane fit's ±2% inlier slab EXCLUDES the true base pixels, so the anchor pools a 6×-wider offset band; (2) **lowest-0.6m band filter** — junction filtering eats base pixels column-wise, and a column whose lowest survivor sits 1m up would poison the gate; coverage feeds the fusion weight instead; (3) **occlusion poison-gate** — a base hidden behind a 1.5m car overshoots ray∩ground by hundreds of metres (t < 1.6×point-depth per point, else refuse); (4) **contamination gate: the anchor REFINES, never TELEPORTS** — street clutter sharing the wall's azimuth donates its own bases, and without |d_geo| ∈ [0.5, 1.5]×|d_depth| a 12m facade "anchors" to the 2m car row (found live on a real street photo — this one does NOT show up in synthetics). Horizon-conditioned fusion (`anchor_horizon_frac`) fades geometric distance toward the depth median for far facades; orientation refit from the footprint line only when it spans ≥ `anchor_min_line_m` with tight lateral fit and within 25° of the cluster azimuth. Roofline segmentation splits a wall cluster at silhouette-height steps (moving-median smoothed) — one plane per building, each re-anchored on its own base; a segment whose re-anchor fails KEEPS the parent cluster's anchor (failed refinement ≠ failed anchor). - **Export-only interior hole fill (`core/mesh_repair.py` + `AtlasExportReliefMesh.fill_interior_holes`, 2026-07-15):** relief meshes are deliberately torn at depth discontinuities so the live 📽 projection never rubber-sheets background onto foreground — that doctrine is load-bearing and untouched. Scope is two composable gates: an edge-count threshold (`max_hole_edges` 64 — the frame perimeter is ~512 edges at grid 128 vs ~4–30 for tears, plus a largest-loop guard) and the artist's **band box** (`fill_depth_near_m`/`fill_depth_far_m`, transcribed off `AtlasBoundedBand.cutoff_m`; a loop fills only if EVERY boundary vertex's forward depth is inside — so the frame, which spans near-to-far, is excluded automatically). **`depth_near_m = 0.0` DISABLES the window** (means "not set"), falling back to threshold-only. **The governing contract is "a fill may leave a hole open, but must never make the mesh worse than not filling"**, and three things follow from it, each found by measurement after the first implementation shipped green on a single convex planar 4-edge fixture: (1) a **fan from `loop[0]` is only valid on a star-shaped loop** — tear footprints are routinely non-convex (L/staircase) and the walk's anchor is arbitrary, so 30% of real fills emitted triangles OUTSIDE the hole over real geometry (ear clipping in the loop's best-fit plane replaced it); (2) **winding is an exact per-edge rule, not a geometric one** — a boundary edge belongs to one face, so the fill sharing it must traverse it the opposite way; comparing the loop's best-fit-plane normal to an adjacent face normal looks equivalent, passes every flat test, and still produced back-facing fills on real (non-planar) holes; (3) **collinear vertices are skipped as ears but never dropped** (dropping leaves a T-junction), and **a diagonal that already exists in the mesh is never reused** (that edge would land in 3 faces = non-manifold) — greedy clipping is scan-order dependent, so every rotation is tried before leaving the hole open. Measured fan→ear-clip on 12 synthetic torn meshes + an end-to-end A/B through the node: fills 30→38, tris-outside 9→**0**, degenerate 2→**0**, non-manifold→**0**, dropped loops 4%→**0**, `is_winding_consistent` True→False becomes True→**True**. ~0.15s on a 31k-face mesh. **Transferable lesson: flat/convex/planar fixtures are the easy case and mesh code passes them by luck — two of the three bugs were invisible to unit tests and only surfaced in an A/B against a real `build_relief_mesh` output**, so `tests/test_mesh_repair.py` now carries `_noisy_relief_mesh` regressions at that level. Purely topological (no learned model, no deps, no Docker) — it caps existing holes, it does NOT predict hidden geometry behind occluders; for that you still need the experimental `AtlasPredictHiddenGeometry` (LaRI/WT). The two are complementary: this repairs what's there, that invents what isn't. **Export-only retopology (2026-07-16, ported from the atlas-ollama research clone):** the same export-only doctrine extended — `retopo_method` (`off`/`quad`/`decimate`/`smooth`, default off) + 4 companion widgets run AFTER the hole-fill on the resolved mesh, before the writers: `quad` = pyinstantmeshes orientation-field quad remesh (BSD wheel), `decimate` = quadric decimation via fast-simplification, `smooth` = trimesh Taubin relax (topology-preserving, UVs kept). quad/decimate change the vertex count, so `core/mesh_retopo.py::regenerate_projective_uvs` REGENERATES the projection-baked UVs by inverting the relief-mesh bake (world → recovered-camera pixel → UV; exact because the bake's rescale-about-cam/floor-clamp/band-clip all move vertices along their own view rays, leaving projected pixels invariant — pinned to 1e-5 in `tests/test_mesh_retopo.py`). The 🔻 retopo result line is appended to the node's on-node report and `preview_solve` carries the retopologized mesh. None of the three deps is a runtime dep; each path raises an actionable ImportError when absent. See `docs/dev/atlas_mesh_retopo.md` + `examples/retopo/`. **Observability (2026-07-15, artist-reported "so the hole filling can't be seen in the viewport?"):** export-only means the fill is invisible in the viewport BY DESIGN — a tear IS a depth discontinuity by construction, so filling one in the LIVE mesh would bridge exactly what the tearing doctrine prevents. But the first cut shipped with the fill count DISCARDED at the call site (the function's own docstring promised "for the node's report" and there was no report), so an artist ticking the box learned nothing short of opening the OBJ in Maya. Now: a `report` output + an on-node `ui.text` render (`web/atlas_export_relief.js`, the `atlas_assess.js` read-only-STRING-widget pattern) naming what filled, what is STILL open (the actionable half — a disappointing fill is usually a too-tight scope, and the outer frame is always one of the remaining loops), and the scope actually applied; plus a `preview_solve` ATLAS_SOLVE output carrying the written mesh, to wire into a viewport. **`preview_solve` deliberately hangs off the EXPORT node rather than a separate preview node** — one set of widgets means the previewed mesh and the exported mesh cannot drift apart, the same reasoning that makes `_resolve_depth_band` shared between `AtlasDepthLayerMask`/`AtlasCleanPlateLayer`. Both outputs are APPENDED (slots 2–3), so saved workflows' existing obj_path/glb_path links are untouched.

- **Inpaint layers (`AtlasDepthLayerMask` + `AtlasCleanPlateLayer` + `ProjectionSource.metadata["projection_mode"]`):** the classic 2.5D matte-painting move — split a solved photo into depth layers, inpaint the regions each layer's foreground occluder hides into a clean plate, then project each clean plate onto its own depth-banded geometry — solving the "orbit off the recovered viewpoint → black holes" limitation (see "Orbit coverage" above) for the SAME camera, unlike `AtlasAddPatchView`'s multi-angle patches which fill gaps via novel AI views at OTHER angles and therefore need angle calibration this approach doesn't. **Deliberately reuses `ProjectionSource` rather than inventing new schema:** the viewport's per-source projection material (see "Camera projection" above) already does everything needed — these two nodes are orchestration only, following the exact `AtlasAddPatchView` pattern minus the orbit (the camera stays the primary's, unchanged, since a clean-plate layer is a same-camera plate, not a novel angle). `AtlasDepthLayerMask` turns one metric depth band into `(layer_mask, occlusion_mask)` — `occlusion_mask` (nearer than the band's near edge) feeds an external `INPAINT_ExpandMask` → `INPAINT_InpaintWithModel` graph to build that band's clean plate; `AtlasCleanPlateLayer` takes the resulting plate image and the same band, clips `build_relief_mesh` to `[near, far]` metres via its new `band_min_m`/`band_max_m` params (the same "exclude the pixel, don't clamp" hole mechanism sky/silhouette exclusion already uses — see "Relief mesh tearing" above), and appends a `ProjectionSource` tagged `metadata["projection_mode"]="clean_plate"`. Both nodes share a private `_resolve_depth_band(metric, valid, near_m, far_m, near_pct, far_pct)` helper (metric depth = `raw_depth * estimate_ground_scale(...)`, same ground-scale path `AtlasDeriveReliefMesh` uses) so their bands can never drift apart — the design requires the mask node's band and the clean-plate node's mesh clip to match exactly, and a shared helper enforces that instead of relying on an artist keeping two nodes' widgets in sync by hand. **The one frontend distinction from patch views:** ordinary multi-angle patches only paint surfaces they see reasonably head-on (`buildPatchSources` passes `facingThreshold: 0.2` to `makeProjectionMaterial`, discarding grazing fragments so other sources can fill them), but a clean-plate layer is a same-camera plate that must paint head-on AND grazing exactly like the primary does — `atlas_blockout.js`'s `buildPatchSources` branches on the serialized `src.projection_mode === "clean_plate"` to pass `facingThreshold: -1` (never facing-discards) instead, relying on depth + `priority` alone (not facing angle) to order overlapping layers; this required threading `projection_mode` through `_extract_blockout_camera`'s `projection_sources` payload, since `ProjectionSource.metadata` was previously dropped entirely during that serialization. **GPL boundary, deliberately kept clean (scope guard):** masking/inpainting itself is never implemented in `atlas_camera` — it's delegated to external ComfyUI node packs wired into the graph (`Acly/comfyui-inpaint-nodes`, GPL-3.0, the `INPAINT_LoadInpaintModel`/`INPAINT_ExpandMask`/`INPAINT_InpaintWithModel` nodes; `scraed/LanPaint`, optional generative KSampler-replacement tier for hard disocclusions a LaMa/MAT pass smears on) — see INSTALL.md's "Optional Inpaint Integration" section. Graph-level composition is not linking, so this doesn't touch Atlas's own license.

- **Camera RAW import (`AtlasLoadRAW` 📷 + `atlas_camera/raw/` + `reference_data/camera_registry.py`, 2026-07-18):** native NEF/CR2/CR3/RAF/ARW support replacing the artist's ACR round-trip, built entirely on existing attachment points: EXIF focal + camera-model→sensor lookup feed `solve_still_image(intrinsics_hint=...)` (the VP node's "0 = auto-detect or EXIF" tooltip, finally implemented) and the NEW `focal_length_mm_hint` on `solve_still_image_learned`/`solve_from_learned_prior`. **EXIF-beats-GeoCalib rule:** a trusted focal REPLACES the prior's focal for fx/fy everywhere the metric cascade looks (reference scale, ground fit, horizon, `estimate_depth(focal_px=)`) while gravity/roll stay GeoCalib's — a joint-estimate approximation, recorded as `debug_metadata["focal_source"]`/`learned_focal_px_predicted`/`focal_disagreement` (warn >25%). Precedence at the node layer (`_resolve_raw_hints`): explicit widget (>0 focal / non-36.0 sensor) beats wired `raw_meta` beats auto — artist override never fights EXIF. **Sensor-size cascade** (mirrors the scale-source tiering, provenance in `sensor_source`): `camera_bodies.json` registry (~34 bodies, `find_camera_body` normalizes messy EXIF strings) → FocalPlane-resolution arithmetic (sanity-clamped 4–70mm) → 35mm-equivalent ratio (`36·focal/focal35`) → assumed 36.0 flagged. **One demosaic per file:** the scene-linear array is the master, the display tensor derives from it, and lensfun undistortion applies ONE shared remap grid to both — so the EXR sidecar and the solve tensor can never disagree geometrically. **Colorspace honesty:** rawpy's linear output has sRGB/Rec.709 primaries; the sidecar is tagged `Linear Rec.709 (sRGB)`, never ACEScg — OCIO converts downstream. **Graceful degrade everywhere:** CR3 metadata is best-effort pure-Python (no ExifTool — misses degrade to manual widgets), lensfun misses are statuses (`no_profile_lens` expected on Fuji X), EXR-codec failures downgrade the plate_ref to proxy with the exact `OPENCV_IO_ENABLE_OPENEXR=1` fix in the report. `raw_meta` inputs on the solve nodes are LINK inputs (not widgets) — saved-workflow-safe by construction; the learned node's `focal_length_mm` widget was APPENDED last per the positional rule. Extras: `[raw]` (rawpy/exifread/opencv<5) + separate `[raw-lens]` (lensfunpy wheels can lag). Pinned by `tests/test_camera_registry.py`, `test_learned_focal_override.py`, `test_load_raw_node.py`; opt-in end-to-end via `ATLAS_TEST_RAW_FILE`. **Live-verified on a real D810 NEF (2026-07-18), which added two rules:** Nikon NEFs carry NO lens-name tag — the lens is derived from MakerNote specs (`_lens_descriptor_from_makernote`, "24mm f/1.4" style; can't distinguish same-spec lenses, so the report names the MATCHED lensfun profile) — and the Lanczos undistort remap overshoots at hard edges (outputs clamped; caught by the integration test's range assert). - **Outlier / stretched-edge tier (ported from the portable-clone worklog `docs/dev/2026-07-18_V1_OUTLIER_STRETCHED_EDGE_WORKLOG.md`, integrated on top of the trust tier 2026-07-18):** three distinct failure classes, three distinct responses — an OUTLIER (hallucinated depth sample) gets detected and EXCLUDED (`AtlasDepthOutlierMask` 🛡: local 3×3 median + robust MAD → explicit holes via the relief nodes' `outlier_mask` input); a STRETCHED EDGE (triangle passes binary tests but is world-space anisotropic) gets whole-quad rejection (`build_relief_mesh(quad_coherence=...)` — both triangles rejected when either fails; node defaults ON for `AtlasDeriveReliefMesh`/`AtlasCleanPlateLayer`/`AtlasDepthLayerMask` (the latter two share the setting so hole-mask QA and layer projection use identical topology), core default OFF for back-compat) plus a MEDIAN-depth world-edge budget (dmax let one bad corner inflate the allowance); a BLACK TEAR means correctly-rejected geometry with no replacement layer — the fix is a deliberate layer (segmented SDXL plate / card / ground / sky dome), NEVER a global threshold raise. Mesh QA stats (`torn_fraction`, `stretch_ratio_p95`, `stretch_fraction_gt12`, `quad_coherence`) ride the relief primitive's metadata into `evaluate_scene_health` per-layer entries with two quality-fallback flags: `stretch_excessive` (p95 > 12) and `torn_excessive` (> 65% — scoped to layers WITHOUT a finite far band, because torn_fraction is global and a deliberately band-clipped layer always reads high; found live when a correct narrow band flagged at 73.8%). The 🩺 gate therefore gates exports on mesh quality automatically. - **Cleanplate-derived hidden support (2026-07-19, canonical OCIO/DCC orbit fix):** plate completion and hidden geometry are separate inference problems. The original depth map has no samples for the road, floor, headland, or facade behind a removed subject. Extending the far side of an `AtlasBoundedBand` with `fill_occluded=True` diffuses depth inward from that far-band boundary; on the ghost-town car and ocean-castle removals this put the support footprint at the cutoff, creating a literal vertical cliff and floating foreground during orbit. The canonical fix is TWO depth solves: original image depth feeds only the explicit SAM/artist-matted foreground layer; the approved full-frame cleanplate gets its own `AtlasDepthMap`, feeding a full-range manual background `AtlasCleanPlateLayer` (`near_pct=0`, `far_pct=0`, `fill_occluded=False`). This lets the cleanplate solver recover continuous support relief instead of inventing a far-cutoff slab. `AtlasBoundedBand` remains valid for restraining a foreground that extrudes too far, and `fill_occluded` remains useful for narrow disocclusion slivers; neither is the default for broad contact/support surfaces. The canonical/marketing ghosttown, oceancastle, and spacehangar graphs are generated by `tools/generate_canonical_ocio_dcc_workflows.py`, stamp `extra.cleanplate_depth_geometry=true`, and were live-run through OCIO → per-layer quadric retopo → Nuke/Maya export. MCP's `atlas://calibration` carries the same rule, and `atlas_inspect_viewport` exposes `n_filled_cells` plus tear/stretch QA so a headless agent can catch regression before export.




- **Live retopology sanctioned via `AtlasRetopologizeLayer` ONLY (2026-07-24, doctrine revision):** the export-only rule above stands for the EXPORT nodes' widgets (they must never mutate the live solve), but the standalone 🔷 node now applies the same `core/mesh_retopo` passes to a chosen layer's serialized relief mesh deliberately. The original objections are resolved, not ignored: (1) the 1:1 vertex-UV invariant is restored exactly by `regenerate_projective_uvs` run with THAT layer's own camera (the bake's ray-preserving property makes the inversion exact, pinned to 1e-5); (2) "smoothing the deliberate tears" is the artist's explicit intent when reaching for the node — it is an opt-in, per-layer act, unlike a global export flag silently reshaping every mesh. Two costs are accepted and reported: a vertex-count change consumes the layer's serialized `edge_risk` coverage field (cleared — the viewport tolerates absence with a harder edge feather on that layer), and quad remesh at high targets can grow the browser payload (`target_vertex_count` is the throttle). Missing optional deps degrade soft with the pip hint in the report — a retopo node must never kill the graph.
